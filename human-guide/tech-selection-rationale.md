# 技術選定の考え方（選定軸の根拠）

このドキュメントは、`new-project-setup-prompts.md` のフェーズ1に焼き込んだ**技術選定軸（Tier 0〜2、A/B の定義）の根拠**をまとめた参照資料である。プロンプト集が「軸の定義（what）」なら、本書は「なぜその定義にしたか（why）」を記録する。

**本書の内容は2種類あり、扱いが異なる**:
1. **選定軸の設計根拠**（§2.1〜2.3・§3・§4 の方法論）——プロンプトに焼き込む正当性を持つ部分。
2. **個別技術への適用例**（§2.4〜2.6 の Rust/TS/Go 分析）——**AIが生成した未検証の論証**であり、ユーザーが根拠を確認・承認した判断ではない。実際の採点はフェーズ1でAIに候補ごとの論証として新たに提示させ、人間が根拠を確認して decision 0001 に記録する。**本書の適用例を"答え"として流用しない**（プロンプト側にも採点結果は焼き込まない——軸に焼き込むのは失敗モードを塞ぐ定義まで）。

前提の文脈: 本プロジェクトは**AI駆動開発の教材**で、要件（ブラウザ動作のBBS）は事前に確定・詳細化されている。技術スタックは**完全に自由**。最終評価は **agent-browser（自然言語でブラウザ操作）による E2E シナリオ**で、空DBから逐次実行される。成功条件は「AIが自律で完成させ、シナリオを通す」こと。

---

## 1. 結論（選定軸の全体像）

**Tier 0 — ハード制約（合否・最適化しない・通すだけ。実際に候補を落とせるもののみ）**
1. 仕様充足（要件・受け入れ基準を満たせる）
2. 評価ハーネス適合（ブラウザ動作／agent-browser操作可／Docker化／永続化／README駆動でsetup・起動／空DBから01→05を逐次・シード無し・非クリア）

**Tier 1 — 主要最適化軸**
- **A. relevant shift-left 被覆率** … このアプリのエラー面のうち、実行時でなくビルド時に検出できる割合
- **B. 残余の反復摩擦** … A で取り返せない純粋な反復コスト

**Tier 2 — 二次/タイブレーク**
- C. エコシステム成熟度・ドキュメント密度（＝AI生成の信頼性）
- D. 運用コスト/性能（イメージ・メモリ・起動・処理速度。GCの実行時コストもここ）
- E. アーキテクチャ等の一般的学習価値
- F. テスト容易性/TDDエルゴノミクス（TDDが背骨なので実質 Tier 1 級）
- G. 再現性のあるビルド（環境依存の非決定性低減）

選定は「複数候補を提示 → 軸で比較 → 推奨提示 → **ユーザーが選択/変更** → decision 0001 に記録（`decided_by: ai+user`）」。

## 2. なぜこの設計か

### 2.1 実現可能性を「二択ゲート」に置かない理由

合否ゲートには「実際に候補を落とせる基準」だけを置く。「AIが完成できるか（実現可能性）」のように全候補が満たしてしまう基準を Tier 0 に置くと**空ゲート**になり、何も落とせない（BBS の CRUD 規模では実現可能性はその典型になりやすい）。これは「実現可能性は全候補が満たす閾値」と言って選定から外すのと実質同じで、意味のある区別を与えない。

正しくは、実現可能性は**二択でなく"程度"**の問題。候補間で実際に差が出るのは「反復コスト（B）」「エコシステム由来のAI生成信頼性（C）」であり、そこで重み付き評価する。落とせる Tier 0 ゲートは「仕様充足」と「評価ハーネス適合」だけに絞る（非ブラウザGUI、コンテナ化困難、永続化なし等は実際に失格にできる）。

### 2.2 A の定義 —「型システムの絶対的強度」ではなく「このアプリで現金化できる早期検出」

A を「型システムがどれだけ強いか」で採点すると、最強クラスの言語（例: Rust）が抽象的に最上位になり、**使わない保証まで加点**してしまう。これが最大の較正ミス。

正しい A は **「このアプリのエラー面（型不一致・null/未処理ケース・SQL/スキーマ不一致・テンプレ不整合 など）を、どれだけビルド時へ寄せられるか」= relevant shift-left 被覆率**。

- 本評価は **agent-browser（ランタイム）** なので、**ランタイム失敗はブラウザ越しデバッグで最も高コスト**。それをコンパイル時へ寄せる価値は特に大きい ⇒ A を Tier 1 に置く理由。
- ただし、静的保証の relevance は**候補・アプリごとの論証**が要る。例えば**メモリ安全（use-after-free）やデータ競合の静的排除**は本物の保証だが、「GC言語には対応する危険が無い／単一スレッドでは競合が起きない／このアプリは共有可変並行を書かない」といった論証が成り立つなら守る対象が無く、A に加点しない（共有インメモリ状態を持つ設計なら加点対象になる）。**この論証が成り立つかどうか自体を、フェーズ1でAIに候補ごとに示させ、人間が確認する**——軸の定義から機械的に「0点」と断定しない。

### 2.3 B の定義 — shift-left と cycles-to-green は違う

混同しやすい2つを厳密に分ける:

- **shift-left** ＝ **本物の欠陥**の検出を実行時→コンパイル時へ移すこと。バグを1個潰す＝**純便益（A）**。
- **cycles-to-green** ＝ **既に正しいコード**をコンパイラに受理させる手間。健全だが不完全な検査器（例: Rustの借用検査）は、実際には安全なプログラムも棄却する。`clone`追加・`Arc`・借用分割等は、バグがあったからではなく「証明可能な形に書き直す」ため＝**捕まえたバグの無い摩擦＝純コスト（B）**。

区別の指針:
> コンパイラが**本物の欠陥**で弾く → shift-left＝A。実行時に払うより安い。
> コンパイラが**既に正しいコード**を「証明できないから」弾く → B。

B が減点する残余コストの指標（順序尺度で相対比較。過度に数値化しない）:

| 指標 | 何を見るか |
|---|---|
| ビルド遅延 | コールド/インクリメンタルのビルド秒数 |
| green までの試行回数 | 型・借用検査を通すまでの edit-compile 反復（cycles-to-green） |
| テスト実行・隔離コスト | 隔離DB供給や起動の重さ（F と連動） |
| 必須の追加儀式 | 変更ごとの codegen/prepare/migration 再生成の負担 |
| ホットリロード/即観測 | 変更を素早く目視できるか |

**ルール**: 「コンパイル段があること／型があること」自体は罰しない（＝A と二重計上しない）。早期検出の便益は A に足し、B は生の遅延・摩擦だけを引く。これで「コンパイル検出＞ランタイム検出」が正しく反映される（A加点・Bは残余摩擦のみ減点）。

> **⚠️ ここから §2.6 までは「個別技術への適用例」＝AIによる未検証の論証。** 技術的主張（ビルドコストの内訳・保証の比較表・「ほぼ同点」等の採点結果）はユーザーが確認・承認したものではない。選定軸をどう適用するかの**論証の形の見本**として読み、フェーズ1では答えとして流用せず、AIに候補ごとの論証を新たに示させて根拠を確認する。

### 2.4 なぜ Rust のビルドは重いのか（各コストが何を買うか）

ビルドの重さの主因と、それぞれが買うもの:

| コストの源 | 仕組み | 何を買うか | BBSで要るか |
|---|---|---|---|
| モノモーフィゼーション | ジェネリクスを型ごとに特殊化コピー生成（C++テンプレート方式）。Goは部分特殊化、TSは型消去 | 実行時速度＋型安全 | 速度分は**ほぼ不要** |
| LLVM最適化(release) | 積極的な機械語最適化。Goは自前の高速バックエンド、TSは最適化なし | 実行時性能 | **不要**（低トラフィック）。dev loopはdebugビルドでこの分は毎反復払わない |
| 借用検査/ライフタイム | MIRベースの所有権解析 | **GCなしのメモリ/データ競合安全** | Go/TSはGCで無料。**GC容認なら不要** |
| トレイト解決/型推論 | コヒーレンス・関連型の解決 | 型安全・抽象化 | 一部有用 |
| proc-macro(sqlx/askama等) | コンパイル時にコード生成・検証を実行 | **SQL/テンプレの正しさをビルド時に検証** | **有用（＝shift-left）** |

要点（AIの主張・要検証）: **Rustのビルドコストの大きい部分（借用検査の摩擦・release最適化・モノモーフィゼーション）が買うのは「GCなしの安全」と「ピーク性能」で、BBSはどちらも要らない**。GCで困らないアプリで借用検査の摩擦を払うのは、必要のない対価。借用検査の主眼（use-after-free・データ競合をGCなしで防ぐ）は、GC前提のGo/TSでは論点にすらならない。

### 2.5 GCコストと並行安全の置き場所

借用検査＋`Send`/`Sync` は**データ競合をコンパイル時に排除**する（Goは実行時検出のみ、TSは単一スレッドで問題を構造上持たない）。Go比では本物の差別化だが、**BBSは共有可変並行を書かない**（各リクエストは概ね逐次、共有はDB経由）ので効く対象が無い。

置き場所:
- **GCの実行時コスト → D**（運用/性能）。差の実際の大きさはアプリ規模に即して論証する（AIの主張: BBS規模では無視できる）。
- **並行安全の便益 → A**。ただし守る対象の有無は要論証（AIの主張: BBSは共有可変並行を書かないため守る対象が無く加点しない。共有インメモリ状態を持つ設計なら加点）。
- **実務的発現 → B**。BBSでのRustの並行機構は、A（捕まるバグ）より、プール/状態を `Arc<AppState>` 等で正しく共有表現する**儀式（B摩擦）**として現れる。

### 2.6 このアプリで"効く"保証は TS/Go でもほぼ取れる

BBS のエラー面で実際にランタイム失敗を潰す保証を、主要候補で比較:

| エラー種 | Rust | TS(strict) | Go |
|---|---|---|---|
| 型不一致・null/未定義・未処理ケース | ◎ | ◎（union＋strict null） | △（nilはコンパイル時に捕まらない・sum型なし） |
| SQL/スキーマ | ◎ sqlx | ◎ Prisma/kysely | ◎ sqlc |
| テンプレート | ◎ askama | ◎ TSX | ×（実行時） |
| メモリ安全(use-after-free) | ◎ | — 起きない | — 起きない |
| データ競合 | ◎ | — 起きない | △実行時検出のみ |

結論（AIの主張・フェーズ1で要検証）: **このアプリの relevant なエラー面では、Rust がコンパイル時に捕まえて TS(strict) が捕まえないものは実質ない**。Rust が余分に捕まえる下2行は、GC/単一スレッド言語では存在しないか BBS が行使しないクラス。したがって A では Rust と TS がほぼ同点、Go は一段下——**という採点はAIの論証であり確定判断ではない**。フェーズ1では表の各行の真偽・網羅性ごと候補ごとに論証させ、人間が確認する。

## 3. よくある較正ミス（避けるべきアンチパターン）

実運用で観測された誤りの型（新プロジェクトの初回実行で発生）:

1. **A の過大評価**: A を「型システムの絶対的強度」で採点し、使わない保証（no-GC安全・ピーク性能）まで加点して最強言語を自動で最上位にする。→ A は relevant 被覆率で採点する。
2. **B の消去**: 反復コスト（借用検査の摩擦・`.sqlx`再生成）を「コストでなく便益／決定論的ガードレール」と言い換えて減点を消す。→ 本物の欠陥を弾くのは A、既に正しいコードを弾くのは B。両建てで正直に数える。
3. **学習価値(E)の密輸**: 所有権を学ぶ教育価値などを正しさ(A)に混ぜる。→ 学びは E、バグ検出は A。軸を分ける。
4. **実現可能性の空ゲート化**: 「全候補が満たす閾値」と称して選定から外す。→ 程度として B・C に吸収する。
5. **退避条項**: 「詰まったら別言語へ」を保険で置く。→ 反復コストを正直に重み付けするなら不要（不信の告白になる）。反復コストの高い案は、その分を A の高さで**明示的に取り返して見せる**。
6. **採点結果の焼き込み**: 軸の定義やプロンプトに個別技術への採点結果（「守る対象が無いので0点」「BBS規模では差は小」等）を焼き込み、候補ごとの論証と人間の確認を省略する。AIの未検証の論証が「確定した根拠」として下流に流れる。→ 軸に焼き込むのは失敗モードを塞ぐ定義まで。採点はAIが候補ごとに論証し、人間が根拠を確認して decision に記録する（本書の初版はこの誤りを含んでいた）。

## 4. 選定時のチェックリスト（この問いに答えられるか）

- この案件で**現金化できる**静的保証はどれか？　その**最小コストの入手先**はどこか？
- A に加点している保証は、**このアプリのエラー面に実在するクラス**を潰しているか？（使わない保証を数えていないか）
- B で引いているのは「捕まえたバグを伴わない摩擦」だけか？　早期検出の便益を二重に減点していないか／cycles-to-green を shift-left と偽っていないか？
- 反復コストの高い候補を採るなら、その対価を **A の高さで明示的に取り返せて**いるか？
- 学習価値(E)・運用コスト(D)・GCコストを、正しさ(A)と混ぜず、それぞれの軸に置けているか？
- A・B の採点は**候補ごとの論証つき**で提示され、その根拠（守る対象の有無・摩擦の実在）を自分（人間）が確認したか？（軸の定義や参照資料の"適用例"から機械的に導いた採点を鵜呑みにしていないか）
- 最終判断は `decided_by: ai+user` になっているか（AIの一存で決めていないか）？
