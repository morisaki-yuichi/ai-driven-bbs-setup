# 研修の目的定義

このドキュメントは `human-guide/` 配下の全文書（プロンプト集・選定軸・各トラック）の**最上流にあたる目的定義**である。下流文書の設計判断が正当かを検証する基準として使い、矛盾が生じた場合は本書を正とする。本書の改訂は人間のみが行う。

制定: 2026-07-19（初回実運用 `../ai-driven-dev-bbs` のフィードバックを受けて）

---

## 1. 実施形態と文書の要求水準

- 実施者は作成者本人（単独）。配布用教材ではない。
- ただし文書の要求水準は配布可能な水準に置く: **手順に任意・オプション・暗黙の前提を残さない（ゆらぎゼロ原則）**。すべてのステップは Must にするか、削除する。
- 根拠: 初回実運用で観測された問題4件（基盤フェーズの git 規律欠落／Lean 任意化による decision ナンバリングずれ／3-B の手順不在／フェーズ3と実装の間の基盤実装フェーズ欠落）は、いずれも「本線の外に置かれた任意・暗黙のステップ」に起因した。単独実施であっても、実行時の判断分岐はコンテキスト再構築コストと手順逸脱を生む。

## 2. 成功の定義

**必須条件（ゲート）**: 評価ハーネス（agent-browser による E2E シナリオ）を通過するアプリの完成。BBS（CRUD）の要求水準は高くないため、これは目的ではなく**学習の測定手段**である。

**主目的**: 以下の3つの学習。

1. **AI駆動開発プロセスの運用**（背骨）: ループ設計、独立レビュー連鎖と正典管理、decision／セッションログ運用、非収束時の打ち切り、AI の見積もり精度の観測（予測バンドと実測ゲート）。全フェーズを貫く学習対象であり、他の2つの学習もこのプロセスの上で行う。
2. **一般的な技術・アーキテクチャ知識**: 認証・セッション設計、migration 規律、層構成（functional core / imperative shell）など、採用技術を通じて得る教養。
3. **形式手法・静的保証の AI 開発における役割**: Lean 形式化と実装の対応、型システムによる shift-left、AI と人間の役割分担（AI が証明を書き、人間が命題・型定義を確認する等）、およびその将来の方向性。

目的2と3は内容面の学習対象として同格。優先が衝突する個別判断は、目的から機械的に導かず人間が決める（decision に記録）。

## 3. 価値指標の定義

「価値」は AI 駆動開発の視点から2系統で定義する。

- **AI ループの効率**: 技術選定軸の A（relevant shift-left 被覆率）・B（残余反復摩擦）・C（生成信頼性）はこちらに属する。shift-left の1バグあたりの価値は「ランタイム側オラクルの弱さ」に比例して重み付ける（テストや agent-browser が最も捕まえにくいバグほど、静的排除の価値が高い）。
- **人間の学習**（主目的2・3）: 技術選定軸の E はこちらに属する。E は「採用技術の一般教養（目的2）」と「形式化・静的保証と実装の対応（目的3）」から成り、**E の重みの根拠は本目的定義である**（実施者の選好ではなく、目的からの導出。初回実運用の decision 0016 がタイブレークに用いた「Lean↔Rust 対応の学習価値」を目的側から支える）。
- ガードレールは維持する: 学習価値（E）を正しさ（A）に密輸しない。

## 4. 本定義からの設計上の帰結

- **Lean 形式化トラックは①（仕様形式化）②（テストオラクル）とも Must。ただし②の「証明の完遂」は必須ではない（2026-07-21 運用知見により改訂）**。①は目的3の主要な学習手段で価値が実証済み（初回運用で15件の decision 起票＋モデルバグ1件発見）。②は「実行可能モデルを実装のテストオラクルに使う」ことを Must とするが、**個々の不変条件の証明完遂はその機能のリスクに見合う場合のみ**とし、非収束したら見送りを裁定する（証明の cycles-to-green 摩擦が実在し、かつ「証明済み」という事実がモデル↔実装ギャップの隠蔽に寄与した実績があるため）。②の背骨は証明でなく**実装側への変異テスト**（§6）に置く。「任意の学習トラック」という位置づけは廃止する。**証明完遂を必須から外した代償として、偽の言明を検出する経路を「証明前反証」（プロパティテスト）で別に張り、モデルと実装の一致は「差分テスト」で機械化する（§7）。**
- **任意ステップの全廃**: 手順書のすべてのステップを Must 化または削除のどちらかに確定させる。3-B（フェーズ3の独立レビュー）は **Must 化**し、他フェーズと同じ「構築→独立レビュー→反映→人間確認」に揃える（2026-07-19 確定）。
- **基盤フェーズ（0〜3）の git 規律**: 各フェーズの人間最終確認後、AI がコミットメッセージを提示して承認を求め、承認後に main へ直接1コミットする（AI が自発的に提案し、ユーザー指示を待たない。2026-07-19 確定）。
- **技術選定軸の改訂**:
  - A の採点ルールを明文化する——Lean オラクルが被覆する面は候補間の A 差に数えない／モデル乖離の検出可能性（網羅マッチ等）は A に数えてよい／モデルと実装の対応の学習価値は E に置く。
  - B は「各候補が実際に採用するループ構成（実在の定番ツールによるベストプラクティス構成）」で採点する。構成の導入・維持コスト自体も B に計上し、推測の最適化による割引は禁止。予測バンドは実測ゲートで検証する。
  - E は §3 のとおり再定義する。
- **モデル↔実装の対応は人手で維持される（機械的に拘束されない）**。Lean 証明が全通しても実装のバグは通過しうる（初回運用: 比較関数を壊しても `lake build` は成功、最大級の SQL バグはモデル対応では捕まらなかった）。したがって検証は**実装（実コード・実 SQL）に対して**行い、モデルへの変異はモデル不変条件の非空性確認にとどまる——両者を混同しない。詳細は §6。
- 具体的な手順改訂は `new-project-setup-prompts.md` ほか各文書で実施し、本書はその根拠を与える。

## 5. 成熟段階（2026-07-19 追記）

研修の運用は2段階で定義する。段階は択一の「任意」ではなく、**学習の進行に対応する順序**である（ゆらぎゼロ原則と矛盾しない）。

- **第1段階 — 手動運用**: 人間が手順書のプロンプトを貼り、全ステップを自ら進行する。目的1（プロセス運用の習得）の学習形態。初回実運用の基盤フェーズはこの形態で完了した。
- **第2段階 — ゲート統治**: オーケストレータAIがプレイブック（`orchestration/`）に従い、モデル指定つきサブエージェントで進行を駆動する。人間は裁定ゲートのみを担う。目的3の「AI と人間の役割分担」の実践形であり、学習対象が「各ステップの運用」から「ゲートの統治」へ一段上がる。
- **適用順序**: 実装フェーズから開始する（反復回数が多く ROI が最大で、テスト・ビルドという強いオラクルがあり進行の失敗が回復可能）。基盤フェーズへの拡張は、次回の新規プロジェクト実行時に検証してから行う。
- **人間ゲートは段階に関わらず人間が担う**: 0-D・0.5-C・スタック選定・レビュー反映の「要確認」・上流エスカレーション承認・非収束エスカレーションは、省力化のためでなく「AIでは代替できない」ことを理由に人間に置かれたゲートであり、オーケストレータに代行させない。
- **実装方式**: Claude Code 内で完結（オーケストレータ＝対話セッション、ワーカー/レビュアー＝サブエージェント）。Agent SDK によるハーネス化は将来の選択肢として残す（本線にはしない）。

## 6. 運用知見: 検証対象の取り違え防止（2026-07-21 追記）

初回運用で観測した最重要の失敗と、その恒久対策。第2段階（ゲート統治）で特に効く。

**何が起きたか**: 実装フェーズのオーケストレーションで、進行役が「Lean 形式モデルへの変異テスト」を「Rust 実装への変異テスト」と取り違えた。セッションログの「変異テスト実施済み」はモデルへの変異であり、実 SQL には対象限定を壊す変異（`where ... or true` ＝全レコード削除、最大級の実害）を検出するテストが存在しないまま複数サイクル生き延びた。ワーカーは自分の認識では正直に「completed」と報告しており、機械可読 STATUS からは不可視。人間ゲート（着手承認・レビュー裁定）も裁定はスコープ・重大度に集中し、「主張した検証が主張した対象を実際に検証したか」を再導出していなかった。

**なぜ構造的か**: これは「AIをフェーズ0〜1へ広げない」判断（§5）と同じ根の別の現れ。オラクルが最強の実装フェーズですら、検証**対象**のカテゴリ混同はオラクルの外側で起き、検出機構がない。したがって拡張ではなく既存工程の設計で塞ぐ。

**恒久対策（各文書に反映）**:
- **検証は実装（実コード・実 SQL）に対して行う。** モデルへの変異はモデル不変条件の非空性確認であって実装の検証ではない。両者を別々に実施する。
- **報告に検証対象を明記させる。** プレイブックの STATUS 契約に `VERIFIED_AGAINST: model | implementation | both` を持たせ、`model` のみの検証を実装の検証と誤認させない。
- **レビューと人間ゲートで検証対象の正当性を再導出する。** 「その検証は主張した対象を本当に検証しているか」をレビュアーの点検項目・G3 のスポットチェックに含める。

## 7. 運用知見: 証明前反証とモデル⇔実装 差分テスト（2026-07-21 追記）

§6 と同じ系列——**オラクルの外側で起き、テストが緑・実装も動く状態に潜む**失敗——の残り2件と、その恒久対策。

**何が起きたか（2件）**:
- **偽の不変条件が通算6件**（`thread_immutable` / `hasNext_iff_more` / `comment_bumps_lastUpdated` / `comment_body_immutable` / `displayName_propagates`——最後の1件は「修正済み」と記録した後も2度穴が残った）。すべて、**証明を試みる過程で**評価器に手作業で反例を食わせて発見した。仕組みが無く、レビュアの手技に依存していた。
- **形式証明は実装を機械的に拘束しない**ことが変異テストで実証された。ソート比較関数を実装側で逆転させても証明系のビルドは成功する。「実装がモデルに従う」保証は人手のレビューにしかなかった。

**なぜ構造的か**: §4 で「証明の完遂はリスク相応（＝必須でない）」に改訂したことにより、**偽の言明を発見する唯一の経路だった「証明を試みて詰まる」が本線から外れた**。証明を見送った言明は、偽であっても何の信号も出さない。したがって反証の網は証明とは独立に張る必要がある。もう一方は、モデルと実装の間に機械的なリンクが1本も存在しないという構造そのもの——モデルへの変異でも実装への変異でも、この対応は検査されない。

**恒久対策（各文書に反映）**:
- **(a) 証明前反証**: 不変条件を立てたら、証明に入る前に decidable な述語としてプロパティテストで反証を試みる。反例が出たら証明せず言明を直す。**証明を見送る言明にも必ず適用する**（見送る場合、これがその言明の唯一のオラクルになる）。反例確定にはカーネル外評価（`native_decide` 等）でなく通常の評価・決定手続きを優先し、信頼ベースを増やさない。
- **(b) モデル⇔実装 差分テスト**: domain の**純粋核**に限り、形式モデルの対応関数と同一入力で出力を照合する。これがモデルと実装の間の唯一の機械的リンク。**Action 層（時刻・乱数・DB・HTTP）は対象外**——非決定性で破綻するうえ、モデルと実装が本質的にずれる（初回運用では「モデルの論理時計は厳密増加だが実装の `now()` は同一トランザクション内で同値になりうる」乖離を decision として明示受容した）。この線引きを守ることが (b) を安く保つ鍵。
- **共通生成器は基盤フェーズで作る（後付け不可）**: 入力・状態の共通生成器／モデル純粋関数を実装側から呼べる経路／両者で一致するシリアライズ・スキーマ、の3点が (a)(b) 共通の土台であり唯一の重い投資。フェーズ4で基盤＋パイロット1本を作り、ループ速度の実測ゲート（`loop-engineering.md` §4.2）と同じ3段階の出口で費用対効果を確認してから実装フェーズで機能ごとに展開する（全機能へ一斉展開しない／機能を積んでから作ると遡及適用で高くつく）。
- **生成器を信頼しない**: プロパティテストが緑でも、生成器が対象空間を張っていなければ「偽の安心」になる。生成器自体を変異検証（意図的に壊した言明・実装を与えて実際に赤くなるか）してから信頼する。
- **(a) は手動の反例検査を置き換えない**: 初回運用は「反例が見つからない＝真」という推論を2回外した。既存の言明・過去に「修正済み」とした言明の再検査は、レビューの人手観点として残す（3度目の再検査で偽が確定した実績がある）。
- **配管を評価者経路に漏らさない**: モデル実行（証明系バイナリの起動）と差分テストは開発 CI に隔離し、評価者のクリーン clone からの起動経路（README 起点の setup・起動）に混ぜない。§6 と同じく再現性不変条件の保護。

**効いているかの測り方（メタ検証）**: 「証明前反証で見つかった偽言明の数」と「差分テストが捕まえたモデル⇔実装乖離の数」をセッションログに計上する。初回運用では前者が手作業で6件、後者は仕組みが無く事後の変異テストで発見した。仕組み化して初回運用を下回るなら、まず生成器の偏りを疑う。

**避けるべきアンチパターン**: ①「証明したから差分テストは不要」——逆。証明はモデル内部の整合しか保証しない。②差分テストを Action 層へ広げる——破綻する。③プロパティテストが緑だから安心する——生成器次第。④(a) を入れたので手動の反例検査をやめる——(a) は反証の網であって真であることの証明ではない。
