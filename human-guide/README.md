# human-guide/ — 人間オペレータ用ガイド（索引）

このディレクトリは、**AI駆動開発を人間が"指揮者(オペレータ)"として回すための資料**を置く場所。**人間が読む**ためのもので、**AIは読み込み・編集しない**。

## このディレクトリの位置づけ（3カテゴリの区別）

役割で分ける:

| ディレクトリ | 役割 | 誰が書く | AIの扱い |
|---|---|---|---|
| `docs/` | **仕様（input）**：作るものの要件・設計・評価基準 | 人 | 正典として**読む**（不可侵） |
| `dev-docs/` | **AIの作業物（output）**：計画・決定記録・セッションログ | AI | **読み書き**する |
| **`human-guide/`（ここ）** | **人間オペレータのメタ手順**：進め方・技術選定/実装原則の根拠 | 人 | **読まない・触らない** |

`human-guide/` はプロジェクトの正典でも作業対象でもない。AIが要件を知るなら `docs/`、作業状態を知るなら `dev-docs/` を見る。

> **AIが読まないための担保**: CLAUDE.md に「`human-guide/` は読み込み・編集しない」除外ルールを明記し、`.claude/settings.json` の `permissions.deny` に `Read(./human-guide/**)` を入れる。実運用でも、手順プロンプトは人間がコピペで貼るもので、AIがこのファイルを開く必要はない。
> **人間は自由に編集してよい**（教材ルールが人に許す「AIへの指示に関するファイル」側の資料）。

「手順書＝何をするか(what)」「根拠2つ＝なぜそうするか(why)」の関係。

## 使い方（最短の流れ）

1. `new-project-setup-prompts.md` を開く。「▶ 説明」は人間向け、その下の**コードブロックだけ**をAIに貼る。
2. 基盤フェーズ **0→1→2→3** を順に進める。各フェーズは「**構築 → 別セッションで独立レビュー → レビュー反映 → 人間が最終確認**」。
3. 基盤完了後は**実装フェーズ**。毎セッション「開始プロンプトで状態復元 → 1機能をTDDで実装 → 運用プロンプトでセッションログ」を繰り返す。
4. スタック選定や実装方針で判断に迷ったら、根拠2ファイルを参照する。

## 前提となる設計思想（1行ずつ）

- **正典チェーンでレビューする**：生specを網羅的に読むのはフェーズ0レビューだけ。以降は上流の確定成果物を正典にし、確定済みフェーズを蒸し返さない（詳細は手順書の前提節）。
- **選定は決め打ちにしない**：複数案＋トレードオフを出し、ユーザーが選ぶ。`decided_by: ai+user` で記録。
- **意図と非決定性をどこで管理するか**が通奏低音（技術選定＝非決定性の制御、実装原則＝意図の配置）。
