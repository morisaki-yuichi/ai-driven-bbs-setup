# Lean による仕様の形式化（Must）

**位置づけ**: 本線の**必須トラック**（`purpose.md` §2 目的3「形式手法・静的保証の AI 開発における役割」の主要な学習手段。旧・任意トラックを 2026-07-19 の目的定義で Must 化）。狙いは2つ——(1) 形式手法（定理証明）を学ぶ、(2) 仕様の曖昧さ・抜けを厳密に炙り出す。AI駆動開発として、人間が AI に Lean の形式化を指揮する形で進める。本書は範囲・境界・不変条件の**根拠（why）**を持ち、実際に貼る手順プロンプトは `new-project-setup-prompts.md`（フェーズ0.5＝0.5-A/0.5-B、実装フェーズ＝実-B）に一元化する。

到達レベルは **(b) 純粋モデル化**（`human-guide/implementation-principles.md` の A/C/D、および技術選定議論の続き）。実 IO・並行の検証(c) や実装抽出には踏み込まない。BBS の要件（有限状態 CRUD ＋明確なルール）は、この (b) にちょうど収まる。

---

## 1. 何をモデル化するか

- **状態**: `Db`（users / threads / comments。comment は `is_deleted` を持つ）。
- **操作**: 登録・ログイン判定・スレッド作成/削除・コメント作成/削除・表示名変更・検索・ソート・ページネーションを、`StateM Db (Except Error α)` の純粋関数として表す。
- **述語**: パスワード強度・表示名長などのバリデーションを decidable な命題として表す。

`StateM Db` は純粋（状態渡しの糖衣）なので、`do` 記法で書いても中身を**証明できる**。モデルは `#eval` で**実行**もでき、実装のテストオラクルに使える。

## 3. 何を証明"しない"か（境界）

- **並行・DB分離レベル・トランザクション競合**（複数ユーザの同時操作）: モデルは逐次・アトミックを仮定。
- **HTTP・セッションCookie・CSRF・argon2 実暗号**: モデルでは抽象化（例: ハッシュは抽象的な単射関数として扱う）。
- **実装（Rust/TS/…）がモデルを refine すること**: 証明しない → **モデルベーステスト＋レビュー**で担保する残余ギャップ。
- これらを本気で検証するなら (c)＝Dijkstra/Hoare モナド・分離論理（Iris）・F\* だが、BBS には過剰。

## 4. 既存方針との接続

- **A/C/D**: 形式化の対象＝**Calculation の核**（バリデーション・認可述語・ソート比較・ページング計算）＋操作の純粋モデル。実 Action（DB/HTTP/セッション）は境界の外＝"imperative shell"。線引きが一致する。
- **shift-left の極北**: エラーを実行時でなく「証明時」に潰す。`tech-selection-rationale.md` のテーゼの延長。
- **decisions と直結**: 形式化は未規定点を必ず明示させる（例: 一覧のコメント数に削除済みを含めるか／論理削除は最終更新日時を更新するか／ソートのタイブレーク）。決めた内容はそのまま `dev-docs/decisions/` に起票する。
- **モデルベーステスト**: Lean モデルを実装のオラクルにして、同じ入力で実装出力と突き合わせる。

## 5. 進め方（Must・2タッチポイント）

形式化の2つの見返り（曖昧さ発見／不変条件の証明・オラクル）は別のタイミングでピークするので、**2つのタッチポイントに分ける**。最後にまとめてやらない。**貼るプロンプトは `new-project-setup-prompts.md` に一元化されている**（本書には複製しない）。

- **タイミング（2タッチポイント）**:
  - **① フェーズ0.5 — 仕様レベルの"軽い"形式化（最大の効果）**: 状態 `Db`・操作・バリデーション述語を書き下し、未規定点（コメント数に削除済みを含むか／論理削除は最終更新日時を動かすか／ソートのタイブレーク等）を露出させる。形式化はスタック非依存なのでフェーズ1を待たない。露出した論点は人間が裁定し、`dev-docs/decisions/` に **0001 からの連番**で起票＋ `requirements-analysis.md` に反映する（decision 連番はここから始まる——スタック decision が固定番号を持たない理由）。**ここでは証明を完遂しない**（目的は抜け出し）。あわせて、②で証明する**不変条件の一覧（機能・評価シナリオとの対応表）を `formal/README.md` に書き出す**——実装フェーズの AI はこのファイルを参照する（`human-guide/` は読めないため、T1〜T8 の対応はここ経由で AI に渡る）。→ プロンプトは 0.5-A / 0.5-B、人間確認は 0.5-C。
  - **② 実装フェーズ — 機能ごとの深い証明＋オラクル（検証の本番）**: 機能を実装する前に、対応する不変条件（T1〜T8）を証明/精緻化する＝通常の Red より一段強い "Red"。実行可能なモデルは実装のテストオラクルにする。機能単位に区切り、全部を一度にやらない。→ 実-B に組み込み済み（証明の非収束はエスカレーションルールの対象）。
  - **原則**: あるルールに依存する機能を実装する前に、そのルールを形式化する（Red-Green-Refactor と噛み合う）。フェーズ2〜4（基盤・評価ハーネス整備・スキャフォールド）では形式化の内容と無関係なのでやらない。
- **置き場所**: 新プロジェクトに **トップレベル `formal/`**（`lake` で独自ビルド）。※これは *AI が書く成果物* なので `human-guide/`（人間専用）とは別扱いで、AI が読み書きしてよい。`docs/`（仕様）・`dev-docs/`（AI作業文書）とも分ける。
- **技術選定への波及**: Must 化により、フェーズ1の A 採点には Lean 採点ルール（モデル被覆面を候補間差に数えない／モデル乖離の検出可能性のみ A／対応の学習価値は E）が適用される。詳細は `tech-selection-rationale.md` §2.7。
