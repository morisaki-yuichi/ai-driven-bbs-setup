# Lean による仕様の形式化（Must）

**位置づけ**: 本線の**必須トラック**（`purpose.md` §2 目的3「形式手法・静的保証の AI 開発における役割」の主要な学習手段。旧・任意トラックを 2026-07-19 の目的定義で Must 化）。狙いは2つ——(1) 形式手法（定理証明）を学ぶ、(2) 仕様の曖昧さ・抜けを厳密に炙り出す。AI駆動開発として、人間が AI に Lean の形式化を指揮する形で進める。本書は範囲・境界・不変条件の**根拠（why）**を持ち、実際に貼る手順プロンプトは `new-project-setup-prompts.md`（フェーズ0.5＝0.5-A/0.5-B、実装フェーズ＝実-B）に一元化する。

到達レベルは **(b) 純粋モデル化**（`human-guide/implementation-principles.md` の A/C/D、および技術選定議論の続き）。実 IO・並行の検証(c) や実装抽出には踏み込まない。BBS の要件（有限状態 CRUD ＋明確なルール）は、この (b) にちょうど収まる。

---

## 1. 何をモデル化するか

- **状態**: `Db`（users / threads / comments。comment は `is_deleted` を持つ）。
- **操作**: 登録・ログイン判定・スレッド作成/削除・コメント作成/削除・表示名変更・検索・ソート・ページネーションを、`StateM Db (Except Error α)` の純粋関数として表す。
- **述語**: パスワード強度・表示名長などのバリデーションを decidable な命題として表す。

`StateM Db` は純粋（状態渡しの糖衣）なので、`do` 記法で書いても中身を**証明できる**。モデルは `#eval` で**実行**もでき、実装のテストオラクルに使える。

## 3. 何を証明"しない"か（境界）

- **並行・DB分離レベル・トランザクション競合**（複数ユーザの同時操作）: モデルは逐次・アトミックを仮定。
- **HTTP・セッションCookie・CSRF・argon2 実暗号**: モデルでは抽象化（例: ハッシュは抽象的な単射関数として扱う）。
- **実装（Rust/TS/…）がモデルを refine すること**: 証明しない → **純粋核については モデル⇔実装 差分テスト（§3.1）で機械化**し、それ以外は**実装への変異テスト＋レビュー**で担保する残余ギャップ。**Lean 証明は実装を機械的に拘束しない**（初回運用: 実装の比較関数を壊しても `lake build` は成功した）。この担保は必ず**実装（実コード・実 SQL）に対して**行う——モデルへの変異はモデル不変条件の非空性確認であって実装の検証ではない。両者の取り違えが初回運用で最大級のバグ（削除の WHERE を壊す変異を検出するテストが実 SQL 側に無いまま複数サイクル生存）を生んだ（背景と恒久対策は `purpose.md` §6）。
- これらを本気で検証するなら (c)＝Dijkstra/Hoare モナド・分離論理（Iris）・F\* だが、BBS には過剰。

### 3.1 残余ギャップに引く2本の機械的な線（Must・2026-07-21 追加）

証明の完遂がリスク相応（＝必須でない）になったことで、上の残余ギャップは2か所で無防備になる。恒久対策の根拠は `purpose.md` §7、貼るプロンプトは `new-project-setup-prompts.md`（フェーズ1・4・実-B）と `orchestration/implementation-playbook.md`（S1〜S3）。

- **(a) 証明前反証**: 不変条件を言明したら、**証明に入る前に** decidable 述語としてプロパティテストで反証を試みる。反例が出たら証明せず言明（仮定の補い方）を直す。塞ぐ穴は2つ——「偽の命題を証明しようとして無限に詰まる」事故と、**証明を見送った言明が偽でも無信号のまま残る**こと。初回運用の偽言明6件は、すべて証明を試みる過程での手作業の反例探索でしか見つからなかった。反例の確定はカーネル外評価（`native_decide` 等）でなく通常の評価・決定手続きを優先する（信頼ベースを増やさない）。
- **(b) モデル⇔実装 差分テスト**: `formal/` の純粋関数と、実装側の対応する純粋関数を**同一入力で出力照合**する。§5 ② の「モデルをテストオラクルにする」を、手技でなく**仕組み**として実体化したもの——両者は別立ての工程ではなく、これが後者の実装形である。入力はランダム生成＋証明で確定した場合分け・境界。
  - **対象は純粋核だけ**。時刻・乱数・DB 状態・HTTP を含む Action 層は対象外（§3 の境界と同じ線）。初回運用では「モデルの論理時計は厳密増加だが実装の `now()` は同一トランザクション内で同値になりうる」乖離を decision として明示受容した——この種のずれは本質的で、差分テストで埋めようとすると破綻する。
  - **共通生成器が唯一の重い投資**。入力・状態の共通生成器／モデルを実装側から呼べる経路（CLI＋JSON 等）／両者で一致するシリアライズ・スキーマの3点は (a)(b) の共有資産で、**フェーズ4（基盤実装）で作る**。機能を積んでから作ると各機能への遡及適用が要る。
  - **生成器自体を変異検証する**。生成器が対象空間を張っていなければプロパティテストの緑は「偽の安心」。意図的に壊した言明・実装を与えて赤くなることを確かめてから信頼する。

**証明・実装への変異テスト・差分テストの守備範囲**（重複しない・互いを代替しない）:

| 手段 | 何を検査するか | 何を検査しないか |
|---|---|---|
| 証明 | モデル内部の整合（言明が真か） | 実装がモデルに従うか |
| 証明前反証 (a) | 言明が偽でないか（反証の網） | 言明が真であること（証明ではない） |
| 差分テスト (b) | 実装の純粋核がモデルと一致するか | Action 層・SQL 層 |
| 実装への変異テスト | テストスイートが実装の欠陥を実際に捕まえるか（差分テスト自身の感度を含む） | 何が正しい振る舞いか |

## 4. 既存方針との接続

- **A/C/D**: 形式化の対象＝**Calculation の核**（バリデーション・認可述語・ソート比較・ページング計算）＋操作の純粋モデル。実 Action（DB/HTTP/セッション）は境界の外＝"imperative shell"。線引きが一致する。
- **shift-left の極北**: エラーを実行時でなく「証明時」に潰す。`tech-selection-rationale.md` のテーゼの延長。
- **decisions と直結**: 形式化は未規定点を必ず明示させる（例: 一覧のコメント数に削除済みを含めるか／論理削除は最終更新日時を更新するか／ソートのタイブレーク）。決めた内容はそのまま `dev-docs/decisions/` に起票する。
- **モデルベーステスト**: Lean モデルを実装のオラクルにして、同じ入力で実装出力と突き合わせる。

## 5. 進め方（Must・2タッチポイント）

形式化の2つの見返り（曖昧さ発見／不変条件の証明・オラクル）は別のタイミングでピークするので、**2つのタッチポイントに分ける**。最後にまとめてやらない。**貼るプロンプトは `new-project-setup-prompts.md` に一元化されている**（本書には複製しない）。

- **タイミング（2タッチポイント）**:
  - **① フェーズ0.5 — 仕様レベルの"軽い"形式化（最大の効果）**: 状態 `Db`・操作・バリデーション述語を書き下し、未規定点（コメント数に削除済みを含むか／論理削除は最終更新日時を動かすか／ソートのタイブレーク等）を露出させる。形式化はスタック非依存なのでフェーズ1を待たない。露出した論点は人間が裁定し、`dev-docs/decisions/` に **0001 からの連番**で起票＋ `requirements-analysis.md` に反映する（decision 連番はここから始まる——スタック decision が固定番号を持たない理由）。**ここでは証明を完遂しない**（目的は抜け出し）。あわせて、②で証明する**不変条件の一覧（機能・評価シナリオとの対応表）を `formal/README.md` に書き出す**——実装フェーズの AI はこのファイルを参照する（`human-guide/` は読めないため、T1〜T8 の対応はここ経由で AI に渡る）。→ プロンプトは 0.5-A / 0.5-B、人間確認は 0.5-C。
  - **② 実装フェーズ — 機能ごとのオラクル＋リスク相応の証明（検証の本番）**: 機能を実装する前に、対応する不変条件（T1〜T8）を**言明**し、実行可能なモデルを実装のテストオラクルにする——ここまでが Must。個々の不変条件の**証明完遂はリスク相応**とし、非収束したら言明のまま見送りを裁定する（2026-07-21 改訂。初回運用で証明の cycles-to-green 摩擦が実在し、`whnf` deterministic timeout の罠を複数回踏んだ／「証明済み」という事実がモデル↔実装ギャップの隠蔽に寄与した、が根拠。`purpose.md` §4）。**②の背骨は証明でなく実装への変異テスト**（§3 の残余ギャップの担保）に置く。機能単位に区切り、全部を一度にやらない。→ 実-B に組み込み済み。
    - **②の内訳（Must の3工程・§3.1）**: 不変条件を言明 → **証明前反証**でプロパティテストにかけ、反例が出たら言明を直す (a) → 純粋関数を実装したら**差分テスト**でモデルと出力照合する (b)。証明完遂だけがリスク相応の任意工程で、(a)(b) は Must。「モデルをテストオラクルにする」の実体が (b) である（別立ての工程ではない）。
  - **共通基盤（フェーズ4）**: (a)(b) が要る共通生成器・モデル呼び出し経路・シリアライズ整合は基盤実装フェーズで作る（後付け不可）。フェーズ1の技術選定では、その導入・維持コストを B に計上する（`tech-selection-rationale.md` §2.7）。
  - **原則**: あるルールに依存する機能を実装する前に、そのルールを形式化する（Red-Green-Refactor と噛み合う）。フェーズ2〜4（基盤・評価ハーネス整備・スキャフォールド）では形式化の内容と無関係なのでやらない。
- **置き場所**: 新プロジェクトに **トップレベル `formal/`**（`lake` で独自ビルド）。※これは *AI が書く成果物* なので `human-guide/`（人間専用）とは別扱いで、AI が読み書きしてよい。`docs/`（仕様）・`dev-docs/`（AI作業文書）とも分ける。
- **技術選定への波及**: Must 化により、フェーズ1の A 採点には Lean 採点ルール（モデル被覆面を候補間差に数えない／モデル乖離の検出可能性のみ A／対応の学習価値は E）が適用される。詳細は `tech-selection-rationale.md` §2.7。

## 6. さらに学ぶ

- Lean 4 公式ドキュメント、"Theorem Proving in Lean 4"、"Functional Programming in Lean"、mathlib。
- Lean のプロパティテスト（Plausible 等）でモデルをオラクル化。**ライブラリの所在・名称は世代交代しうる**（Lean 圏の SlimCheck→Plausible の変遷など）。証明系・実装言語ともプロパティテストライブラリ名は記憶で書かず、初回実装時に一次資料（context7 等）で確認する。
- **スタックが変わる場合の読み替え（§3.1 の抽象化）**: (a)(b) は3軸に分解できる——①モデルが実行可能な証明系（Lean / Isabelle / Coq / Dafny…。抽出やコード生成があると差分テストのレベルが上がる）、②実装言語のプロパティテストライブラリ（proptest / QuickCheck / Hypothesis…）、③両者間のシリアライズ（JSON 等でスキーマ一致）。不変の原則は**純粋核限定・共通生成器・証明前反証・実装⇔モデル差分照合・評価経路隔離・生成器の変異検証**の6つで、これはスタックに依存しない。
- 別味の比較: **TLA+ / Alloy**（軽い反例探索・仕様の無矛盾確認に強い）、**Dafny / F\***（実装抽出・効果検証に強い）。「仕様の抜けを安く見つける」だけなら Alloy/TLA+ の方が ROI が高いことも。
