# ループエンジニアリング（反復ループの設計）

このドキュメントは、AI駆動開発を「反復ループの設計」として捉える観点をまとめた参照資料である。`tech-selection-rationale.md`（選定軸の根拠）の姉妹編として、`new-project-setup-prompts.md` の各仕組みが**ループ設計として何をしているか**を整理し、この観点から追加・検討・見送りした要素の判断を記録する。

用語について: 「ループエンジニアリング」は「ハーネスエンジニアリング」以上に定義が固まっていない新興の言葉で、論者により範囲が揺れる。本書では中核概念——**進捗の単位は1回の優れた応答ではなくループであり、ループそのものを設計対象にする**——に限定して使う。

**ハーネスとの対比**: ハーネスエンジニアリング＝AIを取り巻く**静的な構造**（ルール・ファイル・権限・フック・正典チェーン）の設計。ループエンジニアリング＝その上で回る**動的な過程**（行動→信号→修正の反復）の設計。両者は補完関係にある。

---

## 1. ループの解剖（5つの設計要素）

AI駆動開発の基本サイクル: AIが行動する（コードを書く）→ 環境が信号を返す（コンパイル・テスト・実挙動・レビュー）→ AIが修正する → 終了条件を満たすまで繰り返す。設計要素は5つ:

1. **オラクル（検証器）** — 何が合否を判定するか。テスト・型検査・E2E・人間・別AI。オラクルの品質がループの収束先を決める: オラクルが甘いと「正しいもの」ではなく「緑に見えるもの」に収束する。オラクルは行動者から独立させる（自分の仕事を自分の文脈で採点させない）。
2. **信号の速度とコスト** — 1周にかかる時間と費用。遅いループは同じ予算で回せる反復回数が減り、レイテンシが大きいと「毎回回す」行動自体が崩れる。
3. **ループの階層** — 安くて速い内側ループ（型検査・単体テスト＝秒単位）で大半の誤りを潰し、高価な外側ループ（結合・E2E・人間レビュー＝分〜時間単位）は頻度を落として回す。
4. **終了条件** — いつ止まるか。「完了の定義（DoD）」だけでなく、**収束しないときの打ち切り**も含む。打ち切りの無いループは発散したときにコストを燃やし続ける。
5. **状態の持ち越し** — ループの実行間（セッション間）で何を記憶するか。何を試して何がダメだったかが消えると、同じ失敗を再走する。

## 2. 失敗モード（3つ）

1. **オラクル攻略（Goodhart化）**: 信号を良くすることが目的化し、テストの削除・スキップ・実装追従の改変で「緑に見える」状態を作る。→ 対策: オラクルの不可侵をルール化（CLAUDE.md「やってはいけないこと」）、テストを先に書く（TDD）ことでオラクルを行動前に固定。
2. **発散**: 同一の失敗に対する試行を延々と繰り返し、コストを燃やしながら場当たりの変更でコードを荒らす。→ 対策: 非収束時のエスカレーションルール（§4）。
3. **高価ループへの過依存**: 何でも E2E・実挙動で確かめようとして1周が重くなり、反復回数が激減する。→ 対策: ループ階層の明示（テスト3層）と、コードの多くを最安ループで検証可能にする実装方針（functional core / imperative shell）。

## 3. 既存の仕組みとの対応表

手順書・CLAUDE.md の主要な設計判断は、名前を使っていないだけでループ設計として読める:

| 既存の仕組み | ループ設計としての意味 |
|---|---|
| TDD（Red→Green→Refactor） | 内側ループの典型。テストを先に書く＝オラクルを行動前に固定し、攻略を防ぐ |
| 単体→結合→E2E の3層 | ループ階層そのもの。安い順に固める |
| functional core / imperative shell | コードの多くを最安のループ（純粋関数の単体テスト）で検証可能にする変換 |
| 選定軸A（relevant shift-left 被覆率） | エラー検出を高価なランタイムループから安いコンパイルループへ移す価値の定量化 |
| 選定軸B（cycles-to-green） | 「捕まえたバグを伴わない反復」＝無駄なループ周回のコスト計上。純然たるループ指標 |
| 「1フェーズ＝レビュー1周・再レビューは回さない」 | 外側ループの終了条件。ドキュメント磨きの発散を打ち切る |
| テストの削除・スキップ・実装追従の改変の禁止 | オラクル攻略の禁止 |
| 独立レビュー（別セッション・正典固定） | オラクルの行動者からの独立 |
| セッションログ／状態復元プロンプト | ループ実行間の状態持ち越し |
| 「AIが詰まって人がデバッグしたらルール/スキルに反映」 | 最外周の改善ループ（ループ自体を改善するメタループ） |

つまり本プロジェクトの基盤は、ループエンジニアリングの大部分を暗黙に実装済みであり、以下の §4〜5 は「残りを明示化するか」の判断記録である。

## 4. この観点から導入した要素

### 4.1 非収束時のエスカレーションルール（導入済み）

- **内容**: 同一の失敗（同じテストが同じ理由で赤のまま／同種のエラーが場所を変えて再発）への修正試行が3回連続で進展しなければ、それ以上試行せず、試したこと・仮説・残る症状を整理して報告し、人間の判断を待つ。CLAUDE.md の恒久ルール（2-A 作成仕様）と実-B プロンプトに反映。
- **根拠**: 失敗モード2（発散）は、AI駆動開発で最も高くつく失敗——数十分同じ壁に体当たりしてコストを燃やし、試行錯誤の残骸でコードを荒らす——であり、打ち切りが「人間が気づいて割り込む」しかない状態を放置しない。「停止して試行と仮説を整理して渡す」という出力形式自体が、状態の持ち越し（要素5）の品質を上げる。
- **「3回」について**: 経験的な既定値で厳密な導出はない。合否ゲートではなく「停止して人間に渡すトリガー」なので、運用しながら人間が調整する初期値として置く。

### 4.2 ループ速度の実測ゲート（導入済み）

**内容**: 選定時（1-A）にBの評価根拠を「予測値」——インクリメンタルビルド・単体テスト一式・型/静的検査のみのコマンド・必須儀式の所要時間——として記録させ、実装フェーズ初回セッションのスキャフォールド後に一度実測して突き合わせる（実-B に反映。一度きり・10分程度）。予測は精密な秒数でなく**桁のバンド**（1秒未満／数秒／数十秒／分単位）で書かせる: 秒単位の予測は偽の精密さであり、ループの行動様式を変えるのは桁の違いで、バンドなら「予測2秒・実測4秒」のような疑似係争も起きない。

**なぜ「恣意的な閾値」にならないか**: 「X秒以上は遅い」という絶対閾値は恣意的で、コンパイル時間が買っているもの（shift-left＝A）を無視して時間だけを罰する二重計上になる。しかし基準点を**絶対値でなく「選定時に受け入れた予測値」に固定**すれば、検査するのは「遅すぎないか」の再判定ではなく「**支払うと合意した価格と実際の請求書が一致しているか**」になる。トレードオフの受け入れ自体は選定時に済んでおり、乖離が問題にするのは decision の**前提の反証**であって採否の蒸し返しではない。

**乖離時の出口は3段階**（スタック再選定は経路に無い）: (a) 同バンド→セッションログに記録のみ／(b) 1バンド悪化→ループ構成の最適化（型検査のみの高速コマンドの常用・テストの部分実行・watch モード等）を行い decision（minor）で記録／(c) 2バンド以上の悪化→対処せず人間に報告して判断を待つ（0001 の前提の反証という新規論点であり、上流エスカレーションと同じ扱い）。

補助基準として、「遅い」の実質的な定義は数値でなく**行動変化**——レイテンシのせいで変更をまとめ打ちしたくなる・毎ステップ回すのをやめる——で判定できる（TDD の背骨を折るのはこの行動変化）。再測定の定期実行はせず、この行動変化を感じたときだけ測り直す。

## 5. 意図的に導入しなかったもの — ループ計測の常時ログ化

「毎セッション、何回の反復で緑になったか・どこで詰まったかを記録し、メタループ（ルール/スキル改善）の材料にする」案は**見送り**。この規模の学習プロジェクトでは記録コストが改善便益を上回り、レビュー収束ルールで戒めた「磨きすぎ」のループ版になる。エスカレーションルール（§4.1）の発火時にだけ試行・仮説の詳細が残る設計で、改善ループの材料としては足りる。見送りの記録も設計の一部としてここに残す。
