# 新プロジェクト 基盤構築プロンプト集

`README.md` と `docs/` 一式（人が用意した仕様）だけが置かれた新プロジェクトで、実装前の**開発基盤**（恒久ルール・決定記録・セッションログ・スキル・フック）を構築し、その後の実装まで回すためのプロンプト集。

各プロンプトは**単体で意味が通る**（前セッションの文脈が無くても動く）。フェーズ間の受け渡しは会話履歴ではなく `dev-docs/` 配下のファイルで行うため、途中でセッションが切れても・**フェーズごとに新規セッションで始めても**続行できる。

**使い方**: 「▶ 説明」は人間向けなので貼らない。その下の**コードブロックの本文だけ**をAIに貼る。各フェーズは「**構築 → 独立レビュー → レビュー反映 → 人間が最終確認 → 承認コミット**」の順（フェーズ0は最終確認に 0-D のチェックリストを使う。フェーズ0.5・4 は検証方式が異なる——各節の説明に従う）。基盤フェーズ **0→0.5（Lean仕様形式化）→1→2→3→4（基盤実装）** を終えたら実装フェーズへ入る。**すべてのセッションの終わりにセッションログを残す**（例外: 独立レビューのセッション（0-B/1-B/2-B 等）は、レビュー成果物 `dev-docs/reviews/` 自体が記録を兼ねるため別途のログは不要）。`<...>` は貼る前に差し替える（ただし 0-A/1-A に埋め込んだセッションログ形式内の `<日付>` 等は**AIが埋める**ものなので差し替えない）。

---

## 前提: ディレクトリ / ログ形式 / レビューの正典チェーン（最初に一度だけ）

AI側の基盤・開発ドキュメントは `dev-docs/` に置く（名前は変えてよいが全プロンプトで揃える）。`docs/` は人が書いた仕様として**読み取り専用**。`README.md` は**リポジトリ直下**にあり `docs/` の中ではない（`docs/` は guide/ product/ evaluation/ のみ）。README.md も原則編集しないが、**完全な不可侵ではない**——評価ハーネスの「README.md 起点で setup・起動」を満たすため、**手順への参照追加に限った最小限の追記**を人間の確認のうえ認める（主にフェーズ3で実施。2-A がこの例外を CLAUDE.md に明文化する。README を完全不可侵にするとハーネス前提と矛盾し要件を満たせない）。想定構成:

```
dev-docs/
├─ requirements-analysis.md  フェーズ0の成果（要件サマリ・制約・未決論点）＝以降の正典
├─ harness-invariants.md     評価ハーネス前提の固定リスト（フェーズ0で書き出し・下流レビューの照合用）
├─ session-log-format.md     セッションログの形式定義（フェーズ0で書き出し・以降のプロンプト/スキルが参照）
├─ foundation-plan.md        基盤構築計画・技術選定・DB戦略（フェーズ1）
├─ workflow.md               git/開発フローの詳細（フェーズ2）
├─ ui-ux-guidelines.md       UI/UX・アクセシビリティ方針（フェーズ2）
├─ decisions/                技術判断の記録（README索引 + TEMPLATE + NNNN-*.md。連番はフェーズ0.5の起票から始まるため、特定 decision は番号でなくスラッグで参照する——例: スタック decision ＝ `*-tech-stack.md`）
├─ reviews/                  別セッションによる独立レビュー結果（<フェーズ>-review.md）
└─ session-logs/             セッションごとの要約ログ（決定記録とは別系統）
```

このほかトップレベルに **`formal/`**（フェーズ0.5で作成する Lean 形式化の成果物。AI が読み書きする領域で、`human-guide/` の AI 除外とは別扱い）を置く。

**基盤フェーズの git 規律（フェーズ0〜4 共通）**: workflow.md はフェーズ2の成果物なので、それ以前は本節が正。各フェーズの人間最終確認が済んだら、**AI がコミットメッセージ（Conventional Commits）を提示して承認を求め、承認後に main へ直接1コミット**する（1フェーズ＝1コミット。レビュー反映の修正も同コミットに含める）。構築・レビュー・反映のセッション中は承認なしにコミットしない。各プロンプトにもこの規律を埋め込んである（新セッションの AI は本節を読まないため）。実装フェーズからは workflow.md（短命ブランチ）に切り替える。

**決定記録 vs セッションログ**（CLAUDE.md にも明記させる）:
- `dev-docs/decisions/` … 「**なぜ**その技術判断をしたか」。永続・再検討の対象。`/log-decision` で記録。
- `dev-docs/session-logs/` … 「その回に**何をどこまで**進めたか」。作業の連続性のため。全フェーズ毎回残す。

**セッションログの形式**（正本はこの節。0-A が同じ形式を `dev-docs/session-log-format.md` として書き出し、フェーズ1以降のプロンプトと /log-session スキルはそのファイルを参照する。プロンプト本文への埋め込みは、ファイルがまだ存在しない時点で実行される 0-A だけ——変更時はこの節と 0-A の2か所を同期）:

```
ファイル名: dev-docs/session-logs/<日付>-<連番>-<スラッグ>.md   例: 2026-07-16-01-phase0-analysis.md

# セッションログ: <日付> #<連番> <タイトル>
- フェーズ: 0 | 0.5 | 1 | 2 | 3 | 4 | 開発
- 今回やったこと:
- 決めたこと（関連 decision 番号があれば併記）:
- 次にやること:
- 未解決事項:
```

### レビューの正典チェーン（重要 — 使い回しレビューをやめる理由）

レビュープロンプトを毎回「人が書いた仕様を1から読む」使い回しにすると、レビュアーが毎回要件を再導出し、**すでに確定したフェーズを蒸し返す**（例: フェーズ2レビューがフェーズ1のスタック選定の根拠を再litigationする）。これを防ぐため、**各フェーズのレビューは"正典（source of truth）"を固定**し、そこだけを基準にする:

| レビュー対象 | 正典（これだけを基準にする） | 生spec（README＋docs/）を読むか |
|---|---|---|
| フェーズ0成果（requirements-analysis.md） | README＋docs/（原典） | **網羅的に読む（唯一のゲート）** |
| フェーズ0.5成果（formal/ モデル＋起票 decision） | requirements-analysis.md（AIレビューは置かず、人間が命題・型定義を直接確認＝0.5-C） | 読まない |
| フェーズ1成果（foundation-plan.md＋スタック decision） | requirements-analysis.md | 読まない（固定文言/ACの一点照合のみ可） |
| フェーズ2成果（CLAUDE.md/スキル/フック/workflow/ui-ux 等） | foundation-plan.md＋確定decision＋requirements-analysis.md | 読まない（同上） |
| フェーズ3成果（評価ハーネス適合） | foundation-plan.md＋確定decision＋harness-invariants.md | 読まない |
| フェーズ4成果（基盤実装コード） | ビルド・起動スモーク・テスト（強いオラクル）＋/code-review。ドキュメント型の独立レビューは置かない | 読まない |

- **生specを網羅的に読むのはフェーズ0レビューだけ**。これがある前提で requirements-analysis.md が「以降の正典」の資格を得る。だからフェーズ0レビューは特に厳密に行い、さらに確定前に**人間が 0-D チェックリストで独立に最終確認**する（フェーズ0のすり抜けは後続レビューでは検出されないため、AIレビューと相関しない検出層を重ねる）。
- **評価ハーネス前提は独立ファイルで二重検査**: フェーズ0で `dev-docs/harness-invariants.md` に評価ハーネス前提の固定リストを書き出し、フェーズ1〜3のレビューは成果物をこのファイルと照合する。生specを読ませずに、評価落ちに直結する前提の欠落だけは下流でも毎回再検査される（矛盾があれば上流エスカレーションに記録）。
- **確定済みフェーズの結論は蒸し返さない**。フェーズN（≥1）のレビューは「Nの成果物が上流の正典から**正しく一貫して導出されているか**」だけを見る（要件充足の再判定ではない）。
- **上流エスカレーション弁**: フェーズN のレビュー中に、確定済み上流（例: requirements-analysis）の**実バグ**に気づいた場合だけは、その場で直さず・蒸し返さず、レビュー結果に「**上流エスカレーション**」として1件別枠で記録する。→ 後述の「修正/上流改訂プロンプト」で人間承認のうえ上流を改訂する。これで（a）確定事項の再litigation禁止と（b）上流の見落とし黙殺の両方を防ぐ。

### レビューの収束ルール（回し過ぎない）

- **1フェーズ＝独立レビュー1周**。重大（仕様違反・下流に伝播する誤り）のみブロック。中/軽はバッチで一括反映 or 保留し、**再レビューは回さない**（計画ドキュメントは無限に磨けるため、回すほど反論ログ化して肥大する）。
- **修正が正しいかの確認**: コード（実装フェーズ）は `/verify`・テストで**必ず検証**。ドキュメントに**AIの再レビューを足すのは逆効果**——人間が一読して「その修正は懸念を解消したか、言葉を足して隠しただけか」を見るのが安くて効く。
- レビューが「決定の**根拠が弱い**」と指摘したら、それはAIに書き直させる指摘ではなく**人間の判断に回す**指摘（事実誤りはAIが直す）。

## フェーズ0 — 仕様の取り込み・理解の検証・ギャップ抽出

### 0-A 構築プロンプト

▶ 説明: 要件は揃っているので「ヒアリング」ではなく「AIに復唱させて理解を検証し、仕様が決めていない点を炙り出す」。成果を `requirements-analysis.md` に残す（会話に流れて消えない＝以降の正典になる）。

```
あなたはこのリポジトリの開発を担うAIエージェントです。このプロジェクトは、ブラウザ上で動作する掲示板(BBS)アプリを「AI駆動開発」で作る教材です。要件・設計・評価基準は、リポジトリ直下の README.md と docs/ 配下（guide/ product/ evaluation/）に用意されています。

重要:
- docs/ 配下と README.md は人が用意した仕様であり、このフェーズでは編集しないでください。
- ただし README.md は完全な不可侵ではありません。評価ハーネスの「README.md 起点でセットアップ・起動」を満たすための最小限の追記（手順への参照追加）が、後続フェーズで人間の確認のうえ認められます。制約を抽出する際は、README.md を「編集禁止」と記録せず、この条件付きの扱いをそのまま記録してください（「編集禁止」と記録するとハーネス前提と矛盾します）。
- AI側のドキュメントは docs/ とは別の dev-docs/ に置きます。無ければ作成してよいです。
- git: コミット/プッシュは行わないでください。このフェーズの成果は、私の最終確認後に私が「確定」と伝えたら、コミットメッセージ（Conventional Commits）を提示して承認を得たうえで main へ1コミットします。

このフェーズでは実装とアプリの技術選定はまだしません。README.md と docs/ をすべて読み、分析結果を dev-docs/requirements-analysis.md に書き出してください。4セクション:

1. 要件の構造化サマリ: 機能一覧／各機能の受け入れ基準(AC)／画面／評価シナリオの対応表。
2. 制約の抽出: 技術仕様・開発ルール・評価前提から、実装が必ず守るべき制約を箇条書きに。特に評価ハーネスの前提（ブラウザで動作する／AIがagent-browserで操作できるUIにする／データ永続化は必須／ローカルPCにミドルウェアがある前提にせずDockerで起動する／README.mdを起点にセットアップと起動ができる／評価は空のDBから開始しシナリオ01→05を逐次実行・シード無し・テスト間でDBを消さない）を漏れなく拾うこと。
3. 仕様が決めておらず、AIが前提を置く必要がある論点の一覧（例: 技術スタック、データモデル、認証セッションの持ち方、即時反映の要否、検索の実装方式 など）と、それぞれの論点。**「仕様上の事実」と「AIの解釈で埋めた空白」を明確に区別して書くこと。**
4. 原典カバレッジ表: README.md と docs/ 配下の**全ファイル**について、1ファイル1行で「どのセクションに反映したか／意図的に対象外とした場合はその理由」を表にする（取りこぼし検出用。行の列挙は実際のディレクトリ一覧と突き合わせて作ること）。

あわせて、上記2.の評価ハーネスの前提の一覧を dev-docs/harness-invariants.md として独立ファイルにそのまま書き出してください（後続フェーズのレビューが成果物との照合に使います）。

書き出したら要点を報告してください。最後に dev-docs/session-logs/ にセッションログ（フェーズ0）を1ファイル手書きで残してください（この時点では専用スキル未作成なので手書き）。形式は次のとおり（<日付> 等はあなたが埋める）:

ファイル名: dev-docs/session-logs/<日付>-<連番>-<スラッグ>.md（例: 2026-07-16-01-phase0-analysis.md）

# セッションログ: <日付> #<連番> <タイトル>
- フェーズ: 0
- 今回やったこと:
- 決めたこと（関連 decision 番号があれば併記）:
- 次にやること:
- 未解決事項:

また、この形式定義（ファイル名規約と見出し構成）を dev-docs/session-log-format.md にも書き出してください。以降のセッションはこのファイルを参照して形式を揃えます。
```

### 0-B レビュープロンプト（別セッション・生specを網羅的に読む唯一のゲート）

▶ 説明: **新しい会話**で使う。フェーズ0成果は以降すべての正典になるので、ここだけは原典を網羅的に照合する。

```
あなたは、このリポジトリの成果物を独立の立場でレビューする担当です。生成時の議論は知らない前提で、白紙の目でレビューしてください。

レビュー対象: dev-docs/requirements-analysis.md（フェーズ0の成果）
このファイルは以降の全フェーズの「正典」になります。よって本レビューは、原典（人が書いた仕様）との網羅照合を行う唯一のゲートです。厳密に行ってください。

手順:
1. 人が書いた仕様を網羅的に読む: リポジトリ直下の README.md と docs/ 配下（guide/ product/ evaluation/ の全ファイル）。次に対象ファイルを読む。
2. 仕様から、あなた自身で要点（機能・AC・制約・未決論点）を独立に再導出し、対象ファイルとの差分を洗い出す。対象ファイルの主張を鵜呑みにしない。
3. 敵対的に点検する:
   - 仕様の読み違い・取りこぼした機能/制約・受け入れ基準(AC)やテストシナリオ01〜05との齟齬。
   - 評価ハーネス前提（ブラウザ動作／agent-browserで操作可能／永続化／ローカルにミドルウェアを前提にしない=Docker／README駆動でsetup・起動／空DBから01→05を逐次・シード無し・テスト間でDB非クリア）の取りこぼし。
   - 「仕様上の事実」と「AIの解釈で埋めた空白」の切り分けが正しいか（事実でないものを事実として書いていないか）。
   - 原典カバレッジ表: 表の行が実際のファイル一覧（README.md＋docs/ 配下の全ファイル）と過不足なく一致しているか。「対象外」とされたファイルの理由は妥当か。
   - dev-docs/harness-invariants.md が存在し、上記の評価ハーネス前提リストと過不足なく一致しているか。
   - 根拠の薄い断定・内部の不整合。

出力: dev-docs/reviews/phase0-review.md に、各指摘を重大度で分類して書き出す:
- 重大: 仕様違反・下流に伝播する誤解／取りこぼし
- 中: 根拠不足・不整合・切り分けの誤り
- 軽: 表現・軽微な補足
問題が無い項目は「問題なし」と根拠付きで書いてよい。成果物ファイルは編集しないこと。書き出したら要点を報告してください。
```

### 0-C レビュー反映プロンプト

▶ 説明: 成果物を**修正する側**（生成側 or 新セッション）で使う。全指摘を鵜呑みにせず取捨選択。フェーズ0は正典を作る最重要フェーズなので、指摘は丁寧に扱う。

```
別セッションが出したレビュー結果を requirements-analysis.md に反映します。

レビュー結果: dev-docs/reviews/phase0-review.md
被レビュー成果物: dev-docs/requirements-analysis.md
根拠: リポジトリ直下の README.md と docs/ 配下（フェーズ0は原典が正典）

手順:
1. レビュー結果・被レビュー成果物・原典を読む。
2. 各指摘を重大度ごとに1件ずつ「反映する／反映しない」判断する。レビューは助言であり鵜呑みにしない——原典に照らして妥当な指摘のみ反映する。重大は必ず対応、中/軽はまとめて反映 or 保留（再レビューは回さない）。
3. 反映: 被レビューファイルを直接修正する。
4. 見送り: 黙って捨てず、理由を添えて報告する。
5. 反映した項目／見送った項目（理由つき）を一覧で報告し、変更点を差分として要約する。最後に dev-docs/session-logs/ の既存ログと同じ形式・ファイル名規約でセッションログ（フェーズ0）を手書きで残してください。
6. git: コミットは私が「確定」と伝えるまで行わない（確定後、コミットメッセージを提示して承認を得てから main へ1コミット）。
```

### 0-D 人間の最終確認チェックリスト（フェーズ0確定前・人間が行う／AIに貼らない）

▶ 説明: フェーズ0成果は以降の全フェーズの正典であり、ここをすり抜けた読み違いは後続レビューでは検出されない（生specを網羅的に読むのはフェーズ0レビューだけ）。0-B のAIレビューと相関しない独立の検出層として、フェーズ0を確定する前に人間が以下を原典と直接突き合わせる（5〜10分。全文精読は不要、抜け・混入の検出に絞る）:

- [ ] 原典カバレッジ表の行が、README.md＋docs/ 配下の**実ファイル一覧**と過不足なく一致している（`ls` の結果と突き合わせるだけでよい）
- [ ] 評価ハーネス前提の固定リスト（ブラウザ動作／agent-browserで操作可能／データ永続化／ローカルにミドルウェアを前提にしない=Docker／README駆動でsetup・起動／空DBから01→05を逐次・シード無し・テスト間でDB非クリア）が、requirements-analysis.md と dev-docs/harness-invariants.md の**両方に全項目**ある
- [ ] 「仕様上の事実」欄から2〜3件を抜き取り、原典に実際にその記述があるか確認する（事実とAIの解釈の混入チェック）

## フェーズ0.5 — 仕様の Lean 形式化（Must）

## 選定軸（この階層と定義に厳密に従う）

Tier 0 — ハード制約（合否・最適化しない・通すだけ。実際に候補を落とせるもののみ）
1. 仕様充足: requirements-analysis.md の全制約・受け入れ基準を満たせる。
2. 評価ハーネス適合: ブラウザで動作しagent-browserで操作可能／Dockerでコンテナ化可能／データ永続化／README起点でsetup・起動が完結／空DBから01→05を逐次（シード無し・テスト間非クリア）で回せる。
※ 合否ゲートには「実際に候補を落とせる基準」だけを置く。どの候補も落とさない基準（例: 「AIが完成できるか」の実現可能性が全候補で成り立つ場合）は空のゲートになるため Tier 0 に置かず、その"程度"の差を下記 B・C で重み付き評価する。

Tier 1 — 主要最適化軸（ゲート通過候補を重み付け）
- A. relevant shift-left 被覆率: **このアプリのエラー面**（型不一致・null/未処理ケース・SQL/スキーマ不一致・テンプレート不整合など）のうち、どれだけを実行時でなくビルド時に検出できるか。**型システムの絶対的強度ではなく、この案件で現金化できる早期検出**で採点する。本評価は agent-browser（ランタイム）なのでランタイム失敗はブラウザ越しデバッグで最も高コスト→ shift-left の価値は大きい。
  - 注: 静的保証（メモリ安全・データ競合の排除など）は、**このアプリに実際に守る対象があるか**を候補ごとに論証してから採点すること。守る対象の有無とその根拠（このアプリが共有可変状態や並行処理を持つか等）を比較表に明示し、私が確認できるようにする。論証なしの断定で加点も0点化もしない。
  - Lean モデル（formal/・フェーズ0.5で作成済み）との採点ルール: (1) モデル＋モデルベーステストで検出できる純ロジックのエラー面は、どの候補でもテスト時に捕まるため**候補間の A 差に数えない**。(2) 実装がモデルから乖離するエラー（場合分けの脱落・エラー種の取りこぼし）を**コンパイル時に**検出できる度合い（エラーADTの写し取り・網羅分岐の強制の有無）は A に数えてよい。(3) モデルと実装の対応そのものの学習価値は E に置く（A に数えない）。
- B. 残余の反復摩擦: A（早期検出の便益）で取り返せない**純粋な反復コスト**だけを減点する。「コンパイル段がある／型がある」こと自体は罰しない（= A と二重計上しない）。減点対象: コールド/インクリメンタルのビルド遅延、**既に正しいコードを証明器に受理させるだけの edit-compile 反復（cycles-to-green。捕まえたバグを伴わない摩擦）**、テスト実行・隔離の重さ、変更ごとの必須儀式（codegen/prepare/migration 再生成）、ホットリロード/即観測の有無。
  - 区別: コンパイラが**本物の欠陥**で弾く＝shift-left＝A。**既に正しいコードを証明できず**弾く＝B。
  - B の採点根拠として、各候補の (a) インクリメンタルビルド（1ファイル変更→再コンパイル） (b) 単体テスト一式 (c) 型/静的検査のみの高速コマンド (d) 変更ごとの必須儀式（種類と頻度）の想定所要時間を、**桁のバンド**（1秒未満／数秒／数十秒／分単位）の予測として比較表に明記する（精密な秒数は書かない）。これはAIの予測であり、フェーズ4（基盤実装）で実測と突き合わせる。採用スタック分はスタック decision の B 採点根拠に残す。
  - B は各候補の**素の構成ではなく、実際に採用するループ構成（実在の定番ツールによるベストプラクティス構成——高速リンカ・watch 実行・型検査のみの高速パス・ビルドのレイヤキャッシュ等）**で採点する。その構成の導入・維持コストは儀式として B に計上する。実在が確認できる定番手法に限り、**推測の最適化で割り引かない**。

Tier 2 — 二次/タイブレーク
- C. エコシステム成熟度・ドキュメント密度（＝モデルの生成信頼性・幻覚の少なさ＝AI駆動での実質的な信頼性）。
- D. 運用コスト/性能（イメージ・メモリ・起動・処理速度。**GCの実行時コストもここ**。差の実際の大きさは本アプリの規模に即して論証する）。
- E. 学習価値: (i) 採用技術の一般教養（層構成・SSR/SPA・認証/セッション設計・migration 規律・所有権等の学び）と、(ii) 形式化・静的保証と実装の対応（formal/ の Lean モデルの型構造・エラーADT・述語を、実装言語の型システムへどれだけ忠実に写せるか）。**正しさ(A)と混ぜない**——学びは学び、バグ検出は A（モデル乖離の検出可能性のみ A 側）。
- F. テスト容易性/TDDエルゴノミクス（隔離テスト・Red-Green の速さ。TDDが背骨なので実質 Tier 1 級）。
- G. 再現性のあるビルド（lockfile・オフラインビルド＝環境依存の非決定性低減。A と同じ非決定性制御ファミリー）。

ガードレール（この罠を踏まない）:
- A は「relevant 被覆率」で採点（絶対強度でない）。学習価値(E)を正しさ(A)に密輸しない。
- B は「捕まえたバグを伴わない摩擦」のみ減点（早期検出の便益は A へ）。
- 誰も落とさない基準を合否ゲートに置かない（程度の差は B・C で評価）。
- B を素の構成で採点しない（デフォルトが保守的なだけの調整可能な候補を過大減点するストローマンになる）。逆に、実在しない緩和策の約束で B を消さない。
- 個別技術への採点は、軸の定義から機械的に導かず、候補ごとの論証（このアプリのエラー面・状態の持ち方に即した根拠）を示して行う。
- **反復コストが高い候補は、その分を A の高さで明示的に取り返して見せること。** 退避条項（「詰まったら別言語へ」等）は置かない（反復コストを正直に重み付けするなら不要）。

1. README 駆動: 「README.md を読んで初回セットアップ」「README.md を読んで起動」という固定プロンプトだけで、前提知識なしに setup・起動が完結する導線か（リンク切れ・暗黙の手順・ローカルミドルウェア前提の混入がないか）。README への追記が「手順への参照追加」の範囲を超えていないか。
2. ハーネス前提の照合: dev-docs/harness-invariants.md の**全項目**とフェーズ3の変更が矛盾・欠落していないか。特に空DBから01→05逐次（シード無し・テスト間非クリア）を満たす DB 初期化・マイグレーション戦略になっているか。
3. スモークテストの妥当性: セッションログに3ケース（偽陰性／偽陽性①／偽陽性②）の結果が揃っているか。ダミーファイル等の一時変更が残っていないか。
4. 凍結事項との整合: スタック decision・foundation-plan の確定内容を改変していないか。

上流エスカレーション: 確定済み上流の実バグに気づいた場合は、その場で直さず「## 上流エスカレーション」節に別枠記録する。

出力: dev-docs/reviews/phase3-review.md に重大度分類（重大/中/軽）で書き出す。問題なしは根拠付きで明記。成果物は編集しない。git: コミットは行わない。書き出したら要点を報告してください。
```

```

## 補足メモ

- **レビューは正典を固定して回す**。生specを網羅的に読むのはフェーズ0レビューだけ（＋確定前に人間が 0-D チェックリストで独立確認）。フェーズ1以降は上流の確定成果物（requirements-analysis→foundation-plan＋スタック decision→…）を正典にし、**確定済みフェーズの結論を蒸し返さない**。上流の実バグに気づいたら「上流エスカレーション」で別枠記録→人間承認のうえ改訂。例外として**評価ハーネス前提だけは** `dev-docs/harness-invariants.md`（フェーズ0で書き出し）**との照合を1-B/2-B/3-Bでも毎回行う**（評価落ちに直結する欠落は下流でも再検査する）。
- **スタック decision はフェーズ1で作成・レビュー・確定**。decision の連番はフェーズ0.5（Lean 論点の起票）から始まるため、スタック decision は固定番号を持たない——プロンプト・文書からは番号でなく**スラッグ（`*-tech-stack.md`）で参照**する。フェーズ2以降のレビューはスタック decision を凍結扱いし、転記の忠実性だけ見る（採否を再litigationしない）。
- **基盤フェーズの git 規律（フェーズ0〜4）**: 各フェーズの人間最終確認後、AI がコミットメッセージ（Conventional Commits）を提示し、承認を得て main へ直接1コミット（1フェーズ＝1コミット。レビュー反映も同コミットに含める）。セッション中は承認なしにコミットしない。実装フェーズからは workflow.md（短命ブランチ）に切り替える。
- **収束ルール**: 1フェーズ＝レビュー1周。重大のみブロック、中/軽はバッチ or 保留、再レビューは回さない。修正の検証は、コードは `/verify`・テストで必ず、ドキュメントは人間の一読で（AIの再レビューを足さない）。根拠の強弱に関わる指摘は人間の判断に回す。
- **選定軸の罠を塞ぐ定義**: A＝このアプリのエラー面の relevant shift-left 被覆率（型システムの絶対強度でない）。B＝A で取り返せない残余の反復摩擦（cycles-to-green を shift-left と偽らない）。学習価値(E)を正しさ(A)に混ぜない。誰も落とさない基準は合否ゲートに置かず B・C の程度で評価。反復コストの高い案は A の高さで明示的に取り返して見せる。退避条項は置かない。B は素の構成でなく**実運用のベストプラクティス構成（導入・維持コスト込み・実在の定番ツールに限る）**で採点する（デフォルト構成のストローマンと、推測の緩和策による B 消しの両方を塞ぐ）。Lean モデル（formal/）が被覆する純ロジック面は候補間の A 差に数えず、モデル乖離の検出可能性のみ A、モデルと実装の対応の学習価値は E（E の重みの根拠は `human-guide/purpose.md` §3）。**軸に焼き込むのは失敗モードを塞ぐ定義まで**——個別技術への採点結果（「守る対象が無いので0点」等）はプロンプトに焼き込まず、AIが候補ごとに論証し人間が根拠を確認する。
- **セッションログは全フェーズで残す**。フェーズ0〜1（0.5含む）は手書き（形式の正本は前提節。0-A が `dev-docs/session-log-format.md` に書き出し、1-A 以降のプロンプト・/log-session スキルはそれを参照。変更時は前提節と 0-A の2か所を同期）、フェーズ2で /log-session を作って以降はスキルで。例外: 独立レビューのセッション（0-B/1-B/2-B 等）は、レビュー成果物（`dev-docs/reviews/`）自体が記録を兼ねるため別途のログは残さない。
- **状態復元**: フェーズごと・機能ごとに新セッションで始めてよい。各フェーズ/実装プロンプトの冒頭で、CLAUDE.md・最新セッションログ・decisions・上流成果物を読ませてから作業に入る。
- **非収束時のエスカレーション**: 同一の失敗への修正試行が3回連続で進展しなければ停止し、試行・仮説・症状を整理して人間に渡す（CLAUDE.md の恒久ルール＋実-B に明記。回数は運用で調整してよい初期値）。発散したループはコストを燃やしコードを荒らすため、打ち切りの設計もループ設計の一部（背景は `human-guide/loop-engineering.md`）。
- **実行モデルの配分（2026年7月時点の判断。モデル世代が変われば見直す）**: 原則は「誤りが伝播し、オラクル（検証機構）が弱い工程ほど強いモデル」（原則の導出は `human-guide/loop-engineering.md` §6。世代交代時はそこから再導出する）。
  - フェーズ0〜1（0.5を含む）: 構築 = Claude Opus 4.8／独立レビュー（0-B/1-B）= Claude Fable 5 を推奨。構築プロンプトは指示密度が高く、指示を文字通り遵守する Opus と整合する（Fable は過剰に手順化されたプロンプトで品質が落ちる特性が公式に文書化されている）。レビューは唯一のゲートかつ再レビューしない設計のため**失敗の検出機構がなく**、最も深い推論と反論性向を持つモデルをオラクル側に置く。構築とレビューを別モデルにすること自体が盲点の非相関化として効く。
  - フェーズ2〜4: 構築 = Claude Sonnet 5／独立レビュー（2-B/3-B）と 4-B の /code-review セッション = Claude Opus 4.8。フェーズ0〜1と異なりレビューの失敗は下流（3-Aスモーク・フェーズ4の実動作・実装での実使用）に検出機構があるため、レビューは「構築より一段上・別モデル」で足りる。
  - 実装フェーズ: 実装 = Claude Sonnet 5 を基本とし、難所（認証・セッション設計・WebSocket 等）のセッションのみ Claude Opus 4.8 へ。主たる検証はテスト・/verify であり、AIレビュー（/code-review・/security-review）は補助——これらを回すセッションは Opus 4.8 にする。**/security-review は Opus 推奨**（セキュリティの見落としはテストが拾いにくいため）。
  - Lean 形式化（Must）: ①仕様の形式化（フェーズ0.5）= Claude Opus 4.8——Lean はモデル化の**忠実性**（命題が仕様を正しく写しているか）を検証できず、露出した論点は正典へ伝播するため上流扱い。人間は証明でなく**命題・型定義**を読んで確認する。②機能ごとの証明・テストオラクル（実装フェーズ）= Claude Sonnet 5 を基本とし、証明の非収束（エスカレーション頻発）なら Claude Opus 4.8 へ昇格——Lean カーネルは完全なオラクルで誤った証明は静かに通過しないため、能力差は反復回数にしか現れない。どうしても通らない難証明は Claude Fable 5 を単発投入する。
  - Haiku 級は基盤フェーズ非推奨（長プロンプトの制約遵守密度が不足）。基盤フェーズは一回きりで総トークン量が小さく、コスト差は僅少。配分は「上流に厚く・下流で節約」が構造的に正しい（逆は誤りの伝播コストを最大化する）。
- **ループ速度の実測ゲート**: 1-A で B の採点根拠を予測バンド（1秒未満／数秒／数十秒／分単位）として記録し、フェーズ4（基盤実装）で実測と突き合わせる（4-A）。基準は絶対閾値でなく「選定時に受け入れた予測」——乖離時の出口は記録のみ→ループ構成の最適化→人間へ、の3段階でスタック再選定は経路に無い（背景は `human-guide/loop-engineering.md` §4.2）。
- **実装の2原則**（CLAUDE.md／workflow／実装プロンプトに反映済み）: ①Action/Calculation/Data を分け、純粋な計算に寄せて副作用を薄く隔離（functional core / imperative shell。決定的な核はTDDと非決定性制御に直結）。②意図の置き場所＝コード:How／テスト:What／コミット本文:Why(decision参照)／コメント:Why・Why-not(Howを繰り返さない)。いずれも強制の法でなく“既定の型”。
- **`dev-docs` の名前**を変える場合は全プロンプトで一括置換する。**`docs/` 不可侵**と**セッションログの別系統化**がこの手順の肝。
