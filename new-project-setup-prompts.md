# 新プロジェクト 基盤構築プロンプト集

`README.md` と `docs/` 一式（人が用意した仕様）だけが置かれた新プロジェクトで、実装に入る前の**開発基盤**（恒久ルール・決定記録・セッションログ・スキル・フック）を構築するためのプロンプト集。

各プロンプトは**単体で意味が通る**ように書いてある（前セッションの文脈が無くても動く）。フェーズ間の受け渡しは会話履歴ではなく `dev-docs/` 配下のファイルで行うため、途中でセッションが切れても続行できる。

**使い方**: 「▶ 説明」はあなた（人間）向けなので貼らない。その下の**コードブロックの本文だけ**をAIに貼る。フェーズ0→1→2→3の順に進め、以後は「運用プロンプト」を毎セッション終了時に使う。**フェーズ0〜3も含め、すべてのセッションの終わりにセッションログを残す**。

---

## 前提: ディレクトリ方針とログの形式（最初に一度だけ確認）

AI側の基盤・開発ドキュメントは `dev-docs/` に置く（名前は変えてよいが、全プロンプトで揃えること）。`docs/` は人が書いた仕様として**読み取り専用**。想定構成:

```
dev-docs/
├─ requirements-analysis.md  フェーズ0の成果（要件サマリ・制約・未決論点）
├─ foundation-plan.md        基盤構築計画・技術選定・DB戦略
├─ workflow.md               git/開発フローの詳細
├─ ui-ux-guidelines.md       UI/UX・アクセシビリティ方針
├─ decisions/                技術判断の記録(README索引 + TEMPLATE + NNNN-*.md)
└─ session-logs/             セッションごとの要約ログ(決定記録とは別系統)
```

**決定記録 vs セッションログの棲み分け**（CLAUDE.md にも明記させる）:
- `dev-docs/decisions/` … 「**なぜ**その技術判断をしたか」。永続・再検討の対象。`/log-decision` で記録。
- `dev-docs/session-logs/` … 「その回に**何をどこまで**進めたか」。作業の連続性のため。フェーズ0〜3・開発フェーズを問わず毎回残す。

**セッションログの形式**（`/log-session` スキル作成前のフェーズ0〜1でも、AIにこの形式で手書きさせる）:

```
ファイル名: dev-docs/session-logs/<日付>-<連番>-<スラッグ>.md   例: 2026-07-14-01-phase0-analysis.md

# セッションログ: <日付> #<連番> <タイトル>
- フェーズ: 0 | 1 | 2 | 3 | 開発
- 今回やったこと:
- 決めたこと(関連 decision 番号があれば併記):
- 次にやること:
- 未解決事項:
```

## フェーズ0 — 仕様の取り込み・理解の検証・ギャップ抽出

▶ 説明: 要件はすでに揃っているので「ヒアリング」ではなく「AIに復唱させて理解を検証し、仕様が決めていない点だけを炙り出す」フェーズ。**成果はファイル(`dev-docs/requirements-analysis.md`)に残す**ので、後から見返し・修正できる（修正方法は下の「修正プロンプト」参照）。実装・技術選定はまだしない。

```
あなたはこのリポジトリの開発を担うAIエージェントです。このプロジェクトは、ブラウザ上で動作する掲示板(BBS)アプリを「AI駆動開発」で作る教材です。要件・設計・評価基準は README.md と docs/ 配下(guide/ product/ evaluation/)に用意されています。

重要:
- docs/ 配下は人が用意した仕様であり、読み取り専用です。一切編集しないでください。
- AI側のドキュメントは docs/ とは別の dev-docs/ ディレクトリに置きます。無ければ作成してよいです。

このフェーズでは実装とアプリの技術選定はまだしません。README.md と docs/ をすべて読み、分析結果を dev-docs/requirements-analysis.md に書き出してください。内容は次の3セクション:

1. 要件の構造化サマリ: 機能一覧／各機能の受け入れ基準(AC)／画面／評価シナリオの対応表。
2. 制約の抽出: 技術仕様・開発ルール・評価前提から、実装が必ず守るべき制約を箇条書きに。特に評価ハーネスの前提(ブラウザで動作する／AIがagent-browserで操作できるUIにする／データ永続化は必須／ローカルPCにミドルウェアがある前提にせずDockerで起動する／README.mdを起点にセットアップと起動ができる／評価は空のDBから開始しシナリオ01→05を逐次実行・シード無し・テスト間でDBを消さない)を漏れなく拾うこと。
3. 仕様が決めておらず、AIが前提を置く必要がある論点の一覧(例: 技術スタック、データモデル、認証セッションの持ち方、即時反映の要否、検索の実装方式 など)と、それぞれの論点。

書き出したら、ファイルの要点を私に報告してください。次にこのセッションを終える際は、dev-docs/session-logs/ に所定の形式でセッションログ(フェーズ0)を1ファイル手書きで残してください（この時点では専用スキルは未作成なので手書きでよい）。
```

## フェーズ1 — 技術選定の提案（複数案）＋基盤構築計画（承認ゲート）

▶ 説明: **ここが「決め打ち」への懸念に対応する箇所**。技術スタックは「完全に自由」なので、AIに1案だけ押し付けさせず、**主要レイヤについて複数の有力案とトレードオフ＋推奨を提示させ、あなたが選ぶ/変える**構造にしている。あなたが判断すべき点も明示させる。承認するまでスタックは確定しない（＝ユーザーの裁量が明確に残る）。

```
まだ実装はしません。技術選定の提案と基盤構築計画を作ります。dev-docs/requirements-analysis.md を読んでから始めてください。

前提:
- 技術スタックは自由に選定してよいが、requirements-analysis.md の制約と評価ハーネスへの適合を選定基準にすること。
- docs/ 配下は編集禁止。AI側の計画・記録はすべて dev-docs/ に置く。

やること:
1. まずチャット上で「技術選定の提案」を提示する（この段階ではまだ決定・記録しない）。主要レイヤ(フロント／バック／DBアクセス／DB／認証方式／実行環境／テスト)ごとに:
   - 有力な候補を2案以上、それぞれの長所・短所・評価ハーネス適合の観点で比較する。
   - AIの推奨案とその理由を1つ挙げる。
   - 私（ユーザー）が特に判断・選好を示したほうがよい論点を「要判断」として明示する（例: バックエンド言語、DB種別、即時反映をWebSocketで入れるか等）。
2. 私が各レイヤについて採用案を選ぶ／変更するのを待つ。私が「推奨どおりでよい」と言った場合のみ推奨案を採用してよい。合意できていないレイヤは確定させない。
3. 合意後、dev-docs/foundation-plan.md を作成し、確定スタックを各レイヤ「採用・理由・却下した代替案」で明記。あわせてデータ永続化とDB初期化/マイグレーション方針(空DBから01→05のE2Eが通る前提)、実装前に作る基盤の一覧とディレクトリ構成を書く。
4. スタック確定は「最重要の決定」として、次フェーズで decision 0001 に記録する前提で計画書に書いておく（この段階では計画書に集約でよい）。

計画書の作成後、内容を私に提示して承認を待つこと。承認前に実装や他ファイルの作成へ進まないでください。最後に、このセッションを終える際は dev-docs/session-logs/ にセッションログ(フェーズ1)を所定の形式で手書きで残してください。
```

## フェーズ2 — 開発基盤一式の構築

▶ 説明: 承認済み計画に沿って、恒久ルール(CLAUDE.md)・決定記録の仕組み・**セッションログの仕組み**・スキル・フックを作る。ここで `/log-session` スキルが用意され、以降のログはこれで残せる。

```
承認済みの dev-docs/foundation-plan.md に従い、実装前の開発基盤一式を構築します。1論理単位ずつ作成し、git commit は私が明示的に指示するまで一切行わないこと。

ディレクトリ方針:
- docs/ 配下(人が書いた仕様)は絶対に編集しない。
- AI側の基盤・開発ドキュメントはすべて dev-docs/ に置く:
  dev-docs/requirements-analysis.md(作成済み) / foundation-plan.md(作成済み) / workflow.md / ui-ux-guidelines.md / decisions/(README+TEMPLATE+NNNN) / session-logs/

作成物:
1. CLAUDE.md(リポジトリ直下・恒久ルール): プロジェクト概要／確定した技術スタック／主要コマンド／TDD(Red→Green→Refactor厳守)／Definition of Done／決定記録ルール／セッションログ記録ルール(フェーズを問わず毎セッション末に残す)／UI-UX必須要件／セキュリティ必須要件(XSS・CSRF・SQLインジェクション対策、パスワードハッシュ、セッション失効)／やってはいけないこと。
   - 開発ルールを恒久ルールとして必ず含める: 人間はソースコード・git・docs/ を編集しない。開発・デバッグ・git操作はすべてAIが行う。AIが詰まって人がデバッグした場合は、原因と解決策のフィードバックを受け、再発防止のためルール/スキル/ドキュメントに反映する。
   - dev-docs/ 配下ファイルの参照方法を明記(例: UI作業の前に dev-docs/ui-ux-guidelines.md を読む、詳細フローは dev-docs/workflow.md を参照)。
2. dev-docs/decisions/ 一式:
   - TEMPLATE.md: frontmatter に id / title / date / importance(critical|major|minor) / decided_by(ai|user|ai+user) / status(accepted|superseded)。本文は「背景・問題／決定／根拠／検討した代替案と却下理由／影響範囲・見直し条件」。
   - README.md: importance の定義と索引(decided_by=ai だけを絞り込める一覧を含む)。
   - 0001: 技術スタックの確定を importance: critical / decided_by: ai+user で記録。
3. .claude/skills/log-decision/: TEMPLATEに沿って、連番採番・frontmatter付与・decisions/README.md 索引更新までを一貫して行う決定記録スキル。引数で importance と decided_by を受ける。
4. .claude/skills/log-session/: セッション終了時に、その回の要約を dev-docs/session-logs/ に1ファイル(ファイル名は <日付>-<連番>-<スラッグ>.md)として作成するスキル。記録内容は「フェーズ／今回やったこと／決めたこと(関連decision番号)／次にやること／未解決事項」。これは決定記録(dev-docs/decisions/)とは別系統の作業ログである点を、スキル説明とCLAUDE.mdの両方に明記する。フェーズ0・1で手書きしたログが既にあれば、それも同じ形式に揃える。
5. dev-docs/workflow.md(ブランチ戦略=main保護・機能ごと短命ブランチ、1論理変更=1コミット、Conventional Commits+関連decision参照、コミットはユーザー承認後)と dev-docs/ui-ux-guidelines.md(状態網羅・送信中/成功/失敗のフィードバック・クライアント+サーバの二重バリデーション・a11y・レスポンシブ)。
6. .claude/settings.json: 安全な定型コマンド(git status/diff/add、選定スタックのテスト・lint・ビルド、docker compose、DBマイグレーション等)の allow。Stopフック(直近コミット以降に実装ソースの変更があるのに dev-docs/decisions/ に新規・更新が無ければ、exit 2 で差し戻して「独自判断があれば /log-decision で記録、無ければ記録不要と明示してから終了」を促す。payload の stop_hook_active が true なら exit 0 で許可し無限ループを防ぐ)。MCP は context7(ライブラリ最新ドキュメント)と Playwright(ブラウザ検証)を有効化。

各ファイル作成後、何を作ったかを簡潔に報告し、次に進んでよいか私の確認を取ってください。最後に、作成した /log-session を使ってこのセッション(フェーズ2)のログを残してください。
```

## フェーズ3 — 評価ハーネス適合の作り込み＆スモーク

▶ 説明: 「ここを外すと評価で落ちる」点を基盤に固定し、最後にStopフックとセッションログが実際に効くか確認する。

```
評価ハーネスに適合するための基盤の作り込みと、フックの動作確認を行います。アプリ本体の実装にはまだ入りません。

1. README駆動のsetup/起動: 評価者は「README.md を読んで初回セットアップせよ」「README.md を読んでアプリを起動せよ」という固定プロンプトでこのプロジェクトを動かす。README.md を起点に辿れる形で、セットアップ手順と起動手順のドキュメント(置き場所は dev-docs/ 配下でよい)を用意する方針を決め、その方針を /log-decision で記録する。README.md への追記が必要な場合は、人が書いた仕様の趣旨を壊さない「手順への参照追加」の範囲にとどめ、判断に迷えば私に確認する。
2. 空DBからの逐次E2E(シナリオ01→05、シード無し、テスト間でDBを消さない)を満たすDB初期化・マイグレーション戦略を、dev-docs/foundation-plan.md に反映する。
3. AIが agent-browser で操作する前提に合わせ、隠し要素へのアクションを強いない・極端に操作性の悪いUIを避ける方針を dev-docs/ui-ux-guidelines.md に明記する。
4. スモークテスト: 意図的に「ソースを1行変更したのに dev-docs/decisions/ に記録しない」状況を作り、Stopフックが差し戻して記録を促すことを確認する。確認後、変更は元に戻す。
5. 最後に /log-session を実行し、この基盤構築セッション(フェーズ3)の要約を dev-docs/session-logs/ に残す。

完了したら、基盤構築フェーズが終わり実装フェーズに入れる旨を報告してください。
```

## 修正プロンプト — フェーズ0（や他の dev-docs 成果物）を直したいとき

▶ 説明: **概念のズレや抜け漏れに気づいたら、いつでもこれで直せる**。成果はすべてファイルなので、AIに該当ファイルを編集させる（人はdocs/を触らないルールなので、修正はAIに指示する形で行う）。`<...>` を差し替えて使う。

```
dev-docs/requirements-analysis.md の内容に修正・追記したい点があります。

- 修正箇所: <どのセクション/項目か>
- 変更内容: <正しい理解・追加したい制約・削除したい前提など>

該当ファイルを直接編集し、変更点の差分を要約して報告してください。これに伴い foundation-plan.md 等の後続ファイルに影響があれば、影響箇所も指摘してください（勝手に確定変更はせず、まず指摘のみ）。
```

## 運用プロンプト — 毎セッション終了時（開発フェーズ以降・繰り返し使用）

▶ 説明: 作業を止めるたびにこれを流すと、要約ログが `dev-docs/session-logs/` に溜まる（決定記録とは別物）。

```
今回のセッションを終了します。/log-session を実行し、フェーズ・今回やったこと・決めたこと(関連する decision 番号を併記)・次にやること・未解決事項を dev-docs/session-logs/ に1ファイルとして記録してください。記録が終わったら内容を要約して報告してください。
```

## 補足メモ

- **セッションログは全フェーズで残す**。フェーズ0〜1は `/log-session` スキル未作成のため所定形式で手書き、フェーズ2以降はスキルで作成。フェーズ2でスキルを作った時点で、手書き分も同形式に揃える。
- **フェーズ0の成果はファイル化**(`dev-docs/requirements-analysis.md`)。会話に流れて消えないので、後から「修正プロンプト」で直せる。
- **フェーズ1は決め打ちにしない**。AIが複数案＋推奨を出し、ユーザーが選ぶ/変えるまでスタックは確定しない。decision 0001 は `decided_by: ai+user` として、あなたの判断が入った記録になる。
- **`dev-docs` の名前**を変える場合は、全プロンプトの `dev-docs` を一括置換すること。
- **`docs/` 不可侵**と**セッションログの別系統化**が、この手順の肝。基盤構築後の実装フェーズは、CLAUDE.md と dev-docs/ の各ドキュメントに沿ってTDDで進める。
```
