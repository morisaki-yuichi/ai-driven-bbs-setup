# 実装フェーズ・オーケストレーション プレイブック（第2段階）

**位置づけ**: `human-guide/purpose.md` §5 の第2段階（ゲート統治）。手動運用（`human-guide/new-project-setup-prompts.md` の実-A/実-B＝第1段階・正本）と同じ内容を、**オーケストレータAIがモデル指定つきサブエージェントで駆動**し、人間は裁定ゲートのみを担う形に写したもの。

- 本ファイルは**AI（オーケストレータ）が読む**。`human-guide/`（AI除外）とは別扱いで、新プロジェクトへは `human-guide/` と同時にコピーし、`permissions.deny` の対象に**入れない**。
- ワーカーのプロンプト本文は手順書の実-A/実-B と**意図的に重複**している（ゲート分割に伴う文言差も意図的）。手順書側を変えたら本ファイルも同期する。
- モデル配分は手順書の補足メモ「実行モデルの配分」（2026年7月時点の判断）から導出。世代交代時はそちらを再導出してから本ファイルを追随させる。
- **オーケストレータ自身のモデル: Claude Opus 4.8**（この起動プロンプトを貼るセッションのモデル）。オーケストレータの失敗——特に規律2（ワーカーへ渡すプロンプトに自分の文脈を混ぜる汚染）や規律3〜4（ゲートの飛ばし・STATUS の見落とし）——は人間から見えにくく検出機構がないため、配分原則「誤りが伝播しオラクルが弱い工程ほど強いモデル」に従い最上位を置く。オーケストレータのターンは起動・検分・停止判断が主で長い生成をしないため、コスト増は僅少。

## 起動プロンプト（人間がオーケストレータのセッションに貼る）

```
あなたはこのリポジトリの実装フェーズを進行するオーケストレータです。orchestration/implementation-playbook.md を読み、その規律・ゲート・ステップ定義に厳密に従って進行してください。まず S1（状態復元・着手提案）をサブエージェントで実行し、結果を提示して私の裁定（G1）を待ってください。
```

## オーケストレータの規律

1. **自分では作業しない**: 各ステップは指定モデルのサブエージェントに委譲する。オーケストレータが行うのは、起動・結果の検分・停止判断・人間への提示のみ。
2. **プロンプトは定義どおり渡す**: ワーカーには該当ステップのプロンプトブロック（`<...>` を差し替えたもの）だけを渡す。自分の要約・推測・他ステップの内容・過去ワーカーの生ログを混ぜない（レビューの独立性と、指示の再現性を守るため）。
3. **ゲートは必ず停止する**: G1〜G4 で人間の裁定を得るまで次のステップを起動しない。裁定を推測・代行しない。
4. **STATUS を検分する**: 各ワーカーは報告末尾に共通の STATUS ブロック（後述）を出す。`completed` 以外は即停止して人間に提示する。STATUS ブロックが無い・壊れている場合も停止する（成功と見なさない）。
5. **git はゲートの後ろ**: ワーカーにコミット・push をさせない（ブランチ作成は可）。コミット・PR は G4 の承認後に S5 で行う。
6. **ワーカーの結果を偽らない**: 実行中・未完了のステップの結果を先取りして報告しない。

## ワーカー共通の出力契約（全ステップのプロンプト末尾に付加する）

```
報告の末尾に、次の形式の STATUS ブロックを必ず出力してください:
STATUS: completed | blocked | escalation
BLOCKERS: なし | <内容を1行ずつ>
DECISIONS_LOGGED: なし | <このセッションで /log-decision した番号>
VERIFIED_AGAINST: model | implementation | both | n/a  （検証を伴うステップで必須。何に対して検証したかを宣言する）
ORACLE_YIELD: 証明前反証=<反例が出た言明数>/<反証を試みた言明数> 差分テスト=<捕まえた乖離の件数>/<書いた差分テスト本数> | n/a
NEEDS_HUMAN: なし | <人間の裁定が必要な事項を1行ずつ>
- escalation は「同一の失敗への修正試行が3回連続で進展しない」（テスト実装とも）に達したときに使い、試したこと・仮説・残る症状を本文に整理すること。Lean 証明の非収束は escalation でなく「言明のまま見送り」を NEEDS_HUMAN に記す（証明完遂はリスク相応であり必須でないため）。
- blocked は前提不足・環境問題など自力で解消できない停止に使う。
- VERIFIED_AGAINST: 変異テスト等の検証をモデルに対して行ったのか実装（実コード・実 SQL）に対して行ったのかを明記する。実装の正しさの主張には implementation または both が要る。model のみで実装の検証を主張しない（検証対象の取り違えを防ぐ・purpose.md §6）。**モデル⇔実装 差分テストは「純粋核がモデルと一致すること」の検証であって、実装単体の正しさ（SQL 層・境界層を含む）の検証ではない**——差分テストを根拠に implementation / both を名乗らない。both を名乗るには実装（実コード・実 SQL）への変異テストが別に要る。
- ORACLE_YIELD: 証明前反証と差分テストがどれだけ実際に不具合を捕まえたかの計上（purpose.md §7 のメタ検証）。0件が続く場合は生成器の偏りを疑う材料になるので、0でも省略せず書く。実施していないステップは n/a。
```

## 人間ゲート

| ゲート | タイミング | 人間が裁定すること |
|---|---|---|
| **G1 着手承認** | S1 の後 | 着手する機能の選択／提示された AC・テスト観点・変更範囲の妥当性／**純粋核と Action 層の切り分け**（差分テストの対象範囲）の妥当性／難所指定（S2 のモデルを Opus 4.8 に上げるか） |
| **G2 エスカレーション裁定** | STATUS: escalation / blocked の発生時（随時） | 方針転換・追加情報・打ち切りの判断。証明の非収束ではモデル昇格（Sonnet 5→Opus 4.8、難証明は Fable 5 単発）もここで指示 |
| **G3 レビュー裁定** | S3 の後 | /code-review・/security-review の中/軽指摘の扱い（反映 or 保留）／NEEDS_HUMAN 事項／**検証対象のスポットチェック**（VERIFIED_AGAINST が implementation/both であること、最大級リスクの操作＝削除・認可などの検証が実装への変異で裏づいているかを1〜2件抜き取り確認。**差分テストを根拠に implementation/both を名乗っていないか**も見る） |
| **G4 完了承認** | S4 の後 | DoD 充足の確認（**変更した純粋関数の差分テストが緑／新規不変条件の証明前反証の実施記録**を含む）とコミット・PR の承認（コミットメッセージは S5 が提示） |

## ステップ定義

### S1 状態復元・着手提案（モデル: Sonnet 5）

```
このセッションでは現状把握と着手提案のみを行います。実装はしません。

1. CLAUDE.md を読む（恒久ルール）。
2. dev-docs/session-logs/ の最新ファイルを読み、前回の「次にやること」「未解決事項」を把握する。
3. dev-docs/decisions/README.md の索引で、これまでの確定事項（特に importance: critical）を把握する。
4. 実装対象の仕様は docs/product/issues/（機能とAC）と docs/evaluation/scenarios/（合格基準）にある。未着手・仕掛かりの機能を確認する。
5. 依存順（認証を先に、その後スレッド/コメント/検索・ソート/プロフィール）を踏まえ、次に着手すべき機能の候補と推奨順を挙げる。
6. 最有力候補について、対象issueのAC・テスト観点・想定する変更範囲・対応する不変条件（formal/README.md の一覧から）を要約する。認証・セッション・WebSocket 等の難所に当たるかの見立ても添える。
7. 最有力候補を「**純粋核**（入力→出力が決定的な部分＝モデルとの差分テストの対象）」と「**Action 層**（時刻・乱数・DB・セッション・HTTP を含む＝差分テストの対象外）」に切り分けた見立てを添える。判断基準は「同じ入力なら、いつ・何回呼んでも同じ出力か」。

git 操作は行わないこと。
```

### S2 証明＋TDD実装（モデル: Sonnet 5。G1 で難所指定があれば Opus 4.8）

G1 で承認された機能名を `<機能>` に差し替える。

```
次の1機能を実装します: <機能>（着手内容は進行役経由で承認済み）

進め方（CLAUDE.md と dev-docs/ のルールに従う）:
- main は保護。feat/<name> の短命ブランチで作業する。git commit は行わない（コミットは人間の承認後に別途行う）。
- Lean 形式化（言明は Must・証明完遂はリスク相応。オラクル化は下の「モデル⇔実装 差分テスト」が担う）: 実装に入る前に、この機能に対応する不変条件（formal/README.md の一覧を参照）を formal/ で言明する。個々の証明完遂はその機能のリスクに見合う場合のみ行い、非収束したら深追いせず言明のまま NEEDS_HUMAN に記す（証明の cycles-to-green 摩擦でコードを荒らさない）。証明で確定した場合分け・境界は実装とテストに反映する。
- 証明前反証（Must）: 不変条件を言明したら、**証明に入る前に** decidable な述語にし、基盤の共通生成器によるランダム入力＋境界入力で反証を試みる。反例が出たら証明に進まず、仮定を補って言明を真の形に直してから先へ進む。**証明を見送る言明にも必ず実施する**（見送る場合、これが唯一のオラクルになる）。反例の確定はカーネル外評価（native_decide 等）でなく通常の評価・決定手続きを優先する。
- モデル⇔実装 差分テスト（Must）: domain の純粋関数を実装したら、formal/ の対応するモデル関数と**同一入力で出力照合する差分テスト**を1本以上書く。入力はランダム生成＋証明で確定した場合分け・境界。これが「モデルをテストオラクルにする」の実体であり、モデルと実装の間の唯一の機械的リンク。**対象は純粋核だけ**——時刻・乱数・DB 状態・HTTP を含む Action 層は対象外（非決定性で破綻し、モデルと実装が本質的にずれる。ずれを受け入れる場合は /log-decision に明示的に記録する）。
- 生成器を信頼しない（Must）: プロパティテスト・差分テストが緑でも、生成器が対象空間を張っていなければ偽の安心になる。今回追加・変更した生成器は、意図的に壊した言明・実装を与えて**実際に赤くなる**ことを確かめてから緑を信頼する。
- 検証対象の取り違え防止（Must・②の背骨）: モデルへの変異テストはモデル不変条件の非空性確認であって実装の検証ではない。**実装の正しさは実コード・実 SQL への変異テストで確認する**（例: 削除の WHERE 条件を壊す変異を入れ、単体・結合の両層のテストが実際に赤くなることを確認してから戻す）。両者を別々に実施し、STATUS の VERIFIED_AGAINST で対象を宣言する。Lean 証明は実装を機械的に拘束しない前提を置く（purpose.md §6）。**差分テストは純粋核とモデルの一致しか見ないので、SQL 層・境界層の担保にはならず、この変異テストを置き換えない。**
- TDD厳守: Red(失敗するテストを先に書く) → Green(最小実装で通す) → Refactor(テストは緑のまま整理)。
- 非収束時のエスカレーション: 同一の失敗への修正試行が3回連続で進展しなければ、それ以上試行せず、試したこと・仮説・残る症状を整理して STATUS: escalation で終了する。
- 実装方針(Action/Calculation/Data): ロジックは純粋な計算(Calculation)に寄せて単体で厚く固め、副作用(Action: DB/セッション/HTTP)は薄く隔離する(functional core / imperative shell)。Data は型で表す。単純さを壊す過剰な層は作らない。
- テストの層: 単体(純粋な計算=ロジック/バリデーションを厚く) → 結合(API+DB=Action) → E2E。対応する docs/evaluation/scenarios/ の受け入れ基準を満たすこと。
- 意図の置き場所: コード=How／テスト=What／コミット本文=Why(decision参照)／コメント=Why・Why-not(Howを繰り返さない)。
- 独自の技術判断をしたら、実装を続ける前に /log-decision で記録する。
- Definition of Done のうち本ステップ分を満たす: テスト緑 → **変更した純粋関数にモデルとの差分テストがあり緑** → **新規に立てた不変条件は証明前反証を実施済み（反例の有無を記録）** → /verify で実挙動を確認 → UIを触るなら dev-docs/ui-ux-guidelines.md のチェックを満たす（agent-browserで操作可能・隠し要素にアクションを強いない）。※ /code-review・/security-review は次のステップで別途実行される。
- 評価ハーネス前提を壊さない: 空DBから 01→05 を逐次実行でき、シード無し・テスト間でDBを消さない状態を保つ。マイグレーションで再現可能にする。**モデル実行（証明系バイナリの起動）や差分テストの依存を、README.md 起点の setup・起動（評価者が踏む経路）に持ち込まない**——開発用のテスト経路・CI 側に隔離したままにする。

実装後、何がどのテストで緑になったか・言明/証明した不変条件・**証明前反証で反例が出た言明の件数と内容**・**差分テストが捕まえたモデル⇔実装乖離の件数**・実装への変異テストの結果・/verify の結果を報告してください。
```

### S3 レビュー（モデル: Opus 4.8・新しいサブエージェント）

```
機能 <機能> の作業差分（feat ブランチの未コミット変更を含む）をレビューします。

1. /code-review を実行する。
2. この機能がユーザー入力を扱う場合は /security-review も実行する。
3. 検証対象の再導出（必須）: S2 の報告で「検証済み」とされた主張について、その検証が主張した対象を実際に検証しているかを独立に確かめる。特に、実装（実コード・実 SQL）の正しさの主張が、モデルへの変異でなく実装への変異で裏づけられているかを再確認する。疑わしい箇所は自分で実装へ変異を入れ、テストが赤くなるかを確かめる。取り違え・裏づけ欠落を見つけたら重大として扱う。
4. 差分テストの有無（必須）: 新規・変更された**純粋関数**（入力→出力が決定的な関数）について、対応する formal/ のモデル関数との差分テストがあるかを確認する。無ければ指摘する。Action 層（時刻・乱数・DB・HTTP）に差分テストが広げられていないかも見る——広げられていたら、非決定性で破綻するため指摘する。
5. 生成器の妥当性（必須）: プロパティテスト・差分テストの生成器が対象空間を実際に張っているか（偏っていないか）を見る。生成器自体の変異検証（意図的に壊した言明・実装で赤くなることの確認）が行われたかを確かめ、行われていなければ自分で試す。**生成器が穴だと「偽の安心」を生み、緑が何も意味しなくなる**ため、ここは必ず見る。
6. 偽の言明の再検査（必須）: sorry のまま残る言明・**過去に「修正済み」とした言明**が偽になっていないかを、証明前反証とは独立に人手の観点で点検する。証明前反証は反証の網であって真であることの証明ではなく、「反例が見つからない＝真」という推論は初回運用で2回外れている（3度目の再検査で偽が確定した実績がある）。
7. 重大の指摘のみ自分で修正し、修正後にテストが緑であることを再確認する。
8. 中/軽の指摘は修正せず、対応の要否の一覧として報告する（NEEDS_HUMAN に含める）。報告末尾で、自分が何に対して検証したかを VERIFIED_AGAINST に記す。
```

### S4 ログ（モデル: Sonnet 5）

```
今回の機能実装セッション群の内容をログにします。/log-session を実行し、フェーズ（開発）・今回やったこと（機能 <機能>・言明/証明した不変条件・**証明前反証で見つかった偽言明の件数**・**差分テストが捕まえたモデル⇔実装乖離の件数**・実装への変異テストの結果・レビュー結果の要点）・決めたこと（関連 decision 番号）・次にやること・未解決事項を dev-docs/session-logs/ に1ファイルとして記録してください。前2つの件数は0でも省略せず書くこと（0が続くことは生成器の偏りを疑う材料になるため）。git 操作は行わないこと。
```

### S5 コミット・PR（G4 承認後。モデル: Sonnet 5）

```
機能 <機能> の作業を確定します。dev-docs/workflow.md に従い、1論理変更=1コミットで feat ブランチにコミットし（ヘッダ=簡潔なWhat／本文=Why・関連 decision を参照）、コミットメッセージを提示のうえ、承認済みの内容で PR を作成してください。push と PR 作成まで行い、URL を報告してください。
```

## 進行フロー（1機能 = 1サイクル）

```
S1 → [G1 着手承認] → S2 → (escalation/blocked なら [G2]) → S3 → [G3 レビュー裁定]
   → S4 → [G4 完了承認] → S5 → 次の機能へ（S1 に戻る）
```

- どのステップでも STATUS が completed 以外なら進行を止めて人間へ。
- 全シナリオ（01→05）対応の機能が完了したら、空DBからの逐次E2E通し確認を人間に提案して終了する。
